## 一、Scheduler 是什么？

Scheduler 是 Kubernetes 控制平面的核心组件之一。它的唯一职责就是：为每个还没有分配节点的 Pod，选择一个最合适的 Node。

```
用户创建 Pod (spec.nodeName 为空)
        │
        ▼
   ┌──────────┐
   │ Scheduler │  ← 监听未调度的 Pod
   └────┬─────┘
        │ 选出最佳 Node
        ▼
  创建 Binding 对象 → apiserver → kubelet 收到通知 → 拉起容器
```

---

## 二、源码在哪？

入口点：`cmd/kube-scheduler/main.go` → `cmd/kube-scheduler/app/server.go`

核心逻辑全部在 `pkg/scheduler/` 下：

```
pkg/scheduler/
├── scheduler.go          # Scheduler 结构体 + New() 构造函数
├── schedule_one.go       # 核心调度循环：scheduleOne()
├── eventhandlers.go      # Informer 事件处理（Pod 加入队列）
├── extender.go           # HTTP 扩展器支持
├── framework/            # 插件框架（最重要）
│   ├── interface.go      # 所有插件接口定义
│   ├── plugins/          # 内置插件实现
│   └── runtime/          # 插件运行时调度引擎
├── internal/
│   ├── cache/            # 节点信息缓存
│   ├── queue/            # 调度优先队列
│   └── heap/             # 堆数据结构
└── profile/              # 调度 Profile 管理
```

## 三、一次调度是怎么发生的？

核心入口是 `schedule_one.go` 中的 `scheduleOne()`，`Run()` 方法会不断循环调用它。整个过程分为两大阶段：

### 阶段一：Scheduling Cycle（同步，阻塞）

1. 从队列取出一个待调度 Pod
2. 根据 `pod.Spec.SchedulerName` 选择对应的 Framework（插件集合）
3. 执行调度：

```
                    ┌─────────────┐
                    │  PreFilter  │  预处理，可缩小候选节点范围
                    └──────┬──────┘
                           ▼
        ┌──────────────────────────────────┐
        │           Filter（并行）          │  对每个节点执行过滤
        │  NodeAffinity / NodeResources /  │  回答：这个 Pod 能不能放在这？
        │  TaintToleration / NodePorts ... │
        └──────────────┬───────────────────┘
                       ▼
              没有可行节点？
              → PostFilter（抢占逻辑）
              → 调度失败，回到队列
                       │
                       ▼ 有可行节点
                ┌────────────┐
                │  PreScore  │  评分前的信息收集
                └─────┬──────┘
                      ▼
        ┌──────────────────────────────────┐
        │           Score（并行）           │  对可行节点打分 0-100
        │  ImageLocality / InterPodAffinity│
        │  NodeResourcesBalanced ...       │
        └──────────────┬──────────────────┘
                       ▼
              选得分最高的节点（平局用 reservoir sampling）
```

选出节点后，同步执行：

4. **Assume**：乐观地在缓存中"假定"Pod 已经绑定到该节点（避免下一个调度循环看到过时资源）
5. **Reserve**：调用 Reserve 插件，锁定资源
6. **Permit**：调用 Permit 插件（可返回 Wait 延迟绑定，用于 gang scheduling 等场景）

### 阶段二：Binding Cycle（异步 goroutine）

7. **WaitOnPermit**：如果 Permit 返回了 Wait，在这里阻塞等待
8. **PreBind**：执行绑定前操作（如 Provision 卷）
9. **Bind**：调用 API Server 创建 Binding 对象，真正把 Pod 绑定到 Node
10. **PostBind**：清理工作

**关键设计**：Binding Cycle 在独立 goroutine 中执行，所以调度完一个 Pod 后，Scheduler 可以立刻开始下一个 Pod 的调度，不用等 Bind 完成。

## 四、插件体系（Framework）

这是 Scheduler 最核心的设计。所有调度逻辑都拆成了插件，通过扩展点（Extension Point）挂载到调度流程中：

| 扩展点 | 何时执行 | 典型用途 |
|--------|----------|----------|
| PreEnqueue | Pod 进入队列前 | 门控检查 |
| QueueSort | 队列排序 | 决定先调度谁（全局只能有一个） |
| PreFilter | Filter 之前 | 预处理数据，缩小候选集 |
| Filter | 对每个节点 | 核心过滤：能不能放？ |
| PostFilter | 没有可行节点时 | 抢占逻辑 |
| PreScore | Score 之前 | 为评分收集额外信息 |
| Score | 对可行节点 | 核心打分：放哪最好？ |
| Reserve | 选定节点后 | 乐观锁定资源 |
| Permit | Bind 之前 | 允许/拒绝/等待 |
| PreBind | Bind 之前 | 如挂载卷 |
| Bind | 实际绑定 | 调 API 创建 Binding |
| PostBind | Bind 完成后 | 清理通知 |

### 内置插件一览

| 插件 | 做什么 | 参与的扩展点 |
|------|--------|--------------|
| NodeAffinity | nodeSelector / nodeAffinity 规则 | PreFilter, Filter, Score |
| NodeResourcesFit | CPU/Memory 是否够用 | PreFilter, Filter |
| TaintToleration | 污点容忍检查 | Filter, PreScore, Score |
| InterPodAffinity | Pod 间的亲和/反亲和 | PreFilter, Filter, PreScore, Score |
| PodTopologySpread | 拓扑分布约束 | PreFilter, Filter, PreScore, Score |
| VolumeBinding | PV/PVC 绑定检查 | PreFilter, Filter, Score, Reserve, PreBind |
| ImageLocality | 节点上已有镜像加分 | Score |
| NodePorts | 端口冲突检查 | PreFilter, Filter |
| DefaultPreemption | 默认抢占逻辑 | PostFilter |
| DefaultBinder | 默认绑定实现 | Bind |
| SchedulingGates | 调度门控 | PreEnqueue, QueueSort |

## 五、核心数据结构

```go
// pkg/scheduler/scheduler.go
type Scheduler struct {
    SchedulingQueue   SchedulingQueue      // 优先队列，待调度 Pod 在这里排队
    Profiles          profile.Map          // SchedulerName → Framework 的映射
    Cache             Cache                // 节点信息缓存（含 assumed pods）
    nodeInfoSnapshot  *Snapshot            // 每轮调度前从 Cache 拍快照
    Extenders         []Extender           // HTTP 扩展器
    client            clientset.Interface  // 访问 API Server
    // ...
}
```

- **SchedulingQueue**：PriorityQueue 实现，支持 active/backoff/unschedulable 三个子队列
- **Profiles**：支持多个调度配置（比如一个给普通 Pod，一个给 GPU Pod），通过 `pod.Spec.SchedulerName` 选择
- **Cache**：维护每个节点的资源使用情况，`AssumePod()` 是乐观更新的关键

## 六、调度队列（PriorityQueue）

实现在 `pkg/scheduler/internal/queue/scheduling_queue.go:145`，包含三个子结构：

| 队列 | 作用 |
|------|------|
| activeQ（堆） | 就绪待调度的 Pod，Pop() 从此取 |
| podBackoffQ（堆，按退避到期时间排序） | 调度失败后等待重试的 Pod |
| unschedulablePods | 被判定为不可调度的 Pod |

**流转逻辑**：
- 新 Pod 由 informer 事件处理器加入 activeQ
- 调度失败 → 进入 backoffQ 或 unschedulablePods
- 集群事件（节点变化、PVC 绑定等）触发时，unschedulablePods 中匹配事件的 Pod 被移回 activeQ
- 退避时间从 1 秒指数增长到 10 秒

## 七、失败处理

如果一个 Pod 调度失败（没有可行节点），Scheduler 会：

1. 记录是哪些插件导致失败的（下次可以跳过）
2. 把 Pod 放回 unschedulable 队列（不会反复重试浪费 CPU）
3. 尝试 PostFilter（抢占）：驱逐低优先级 Pod 来腾出空间
4. 给 Pod 打上 `PodScheduled=False` 的 condition
5. 发出 `FailedScheduling` Event

当集群状态变化（某个 Pod 被删除、节点资源释放）时，`eventhandlers` 会把之前失败的 Pod 重新放回 active 队列。

## 八、调度周期详解

schedulingCycle 按顺序执行以下步骤：

### 1. PreFilter

准备本轮调度所需的全局状态，可以缩小待评估的节点范围。例如 NodeResources 插件在这里计算 Pod 的资源请求总和。

### 2. Filter（原名 Predicates）

对每个候选节点并行执行可行性检查。通过 Filter 的节点称为"可行节点"（feasible nodes）。

常见 Filter 插件：
- **NodeResourcesFit** — 节点资源是否足够
- **NodeAffinity** — 是否满足 nodeAffinity 规则
- **TaintToleration** — Pod 能否容忍节点的 taint
- **InterPodAffinity** — Pod 间亲和/反亲和
- **PodTopologySpread** — 拓扑分布约束

### 3. PostFilter（仅在 Filter 全部失败时触发）

主要用途是抢占（Preemption）——尝试驱逐低优先级 Pod 来腾出空间。

### 4. PreScore

为打分阶段做信息准备，比如计算拓扑分布信息。

### 5. Score（原名 Priorities）

对可行节点并行打分（0-100），经过 NormalizeScore 归一化后按权重加总，选出得分最高的节点。

常见 Score 插件：
- **NodeResourcesBalancedAllocation** — 均衡资源分配
- **ImageLocality** — 偏好已有镜像的节点
- **InterPodAffinity** — Pod 间亲和偏好

### 6. Assume

乐观缓存：在实际 Bind 完成之前，就先在调度器缓存中记录"该 Pod 已绑定到某节点"，让后续调度循环不必等待 API 写入。

### 7. Reserve

插件预留资源（如 VolumeBinding 预留存储卷），与 Unreserve 配对用于失败回滚。

### 8. Permit

审批/拒绝/等待。返回 Wait 则 Pod 会阻塞在 Permit 阶段（可用于 gang scheduling 场景）。

---

## 九、绑定周期（bindingCycle，异步执行）

1. **WaitOnPermit** — 等待 Permit 插件放行
2. **PreBind** — 绑定前准备（如预配卷）
3. **Bind** — 调用 API Server 写入绑定（默认用 DefaultBinder）
4. **PostBind** — 清理/通知钩子

任何步骤失败都会触发 Unreserve 回滚预留的资源。

---

## 十、一句话总结

> Scheduler 本质上就是一个无限循环：从队列取 Pod → 拍节点快照 → 过滤 → 打分 → 绑定。所有具体逻辑都通过插件框架注入，可插拔、可扩展。

> 整个设计的核心思想是：Filter 做减法（排除不可行），Score 做加法（选出最优），插件化架构保证可扩展性，异步绑定提高吞吐。
