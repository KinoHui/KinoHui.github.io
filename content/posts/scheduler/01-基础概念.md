## 前言

在学习 Kubernetes Scheduler 之前，我们需要先理解几个基础概念：Condition、Event 和 Binding。这些概念贯穿整个 K8s 资源生命周期，是理解调度器行为的基础。

---

## Condition（条件）

Condition 表示资源当前状态的一个维度/方面。它回答的是"这个资源的某个方面现在是什么情况？"

### 结构定义

```go
// staging/src/k8s.io/apimachinery/pkg/apis/meta/v1/types.go:1481
type Condition struct {
    Type               string          // 条件类型，如 "Available", "Ready"
    Status             ConditionStatus // 取值: True / False / Unknown
    ObservedGeneration int64           // 基于哪一代 spec 观察到的
    LastTransitionTime metav1.Time     // 上次状态变化的时间
    Reason             string          // 机器可读的原因（CamelCase）
    Message            string          // 人类可读的详细描述
}
```

### 核心字段解释

| 字段 | 含义 |
|------|------|
| Type | 条件的种类。不同资源有不同的 condition type，如 Pod 有 Ready、Initialized、ContainersReady、PodScheduled |
| Status | 三值逻辑：True（满足）、False（不满足）、Unknown（无法确定） |
| Reason | 短的机器可读字符串，解释为什么是这个状态，如 ContainersNotReady、Unschedulable |
| Message | 人类可读的详细信息，适合出现在 kubectl describe 输出中 |
| LastTransitionTime | 状态发生变化的时间点（不是上次检查的时间） |

### Pod Condition 示例

```bash
kubectl get pod my-pod -o jsonpath='{.status.conditions}' | jq
```

输出示例：

```json
[
  {
    "type": "Initialized",
    "status": "True",
    "reason": "ContainersInitialized"
  },
  {
    "type": "Ready",
    "status": "False",
    "reason": "ContainersNotReady",
    "message": "containers with unready status: [my-container]"
  },
  {
    "type": "ContainersReady",
    "status": "False",
    "reason": "ContainersNotReady"
  },
  {
    "type": "PodScheduled",
    "status": "True",
    "reason": "SchedulerSuccess"
  }
]
```

### Condition 的设计哲学

- **声明式**：Condition 反映的是"当前实际状态"，不是期望状态
- **多维度**：一个资源可以有多个 Condition，分别描述不同的方面
- **可聚合**：上层工具（如 kubectl、HPA）通过读取 Condition 来判断资源健康状况
- **merge 策略**：Condition 使用 `patchMergeKey: "type"`，意味着按 Type 做合并更新，不会重复添加

## Event（事件）

Event 是集群中发生的离散事件的日志记录。它回答的是"发生了什么事？"

### 结构定义

```go
// staging/src/k8s.io/api/core/v1/types.go:6036
type Event struct {
    metav1.TypeMeta
    metav1.ObjectMeta

    InvolvedObject     ObjectReference // 事件关联的对象（哪个 Pod/Node/...）
    Reason             string          // 机器可读的原因
    Message            string          // 人类可读的描述
    Source             EventSource     // 报告事件的组件（如 kubelet, scheduler）
    FirstTimestamp     metav1.Time     // 首次记录的时间
    LastTimestamp      metav1.Time     // 最近一次发生的时间
    Count              int32           // 同一事件重复发生的次数
    Type               string          // Normal 或 Warning
    ReportingController string         // 发出事件的控制器名
    ReportingInstance   string         // 控制器实例 ID
}
```

### Event vs Condition 的关键区别

| 维度 | Condition | Event |
|------|-----------|-------|
| 本质 | 持续状态的快照 | 离散的时间点记录 |
| 生命周期 | 长期存在，跟随资源 | 有保留时间（默认1小时），会被清理 |
| 回答的问题 | "现在怎么样？" | "发生了什么？" |
| 数据模型 | 类型+状态 三值逻辑 | 时间戳+原因+消息 |
| 存储 | 资源的 `.status.conditions` 字段 | 独立的 Event 资源对象 |
| 典型来源 | Controller 观察后设置 | 组件运行时产生的日志 |

### Event 的实际表现

```bash
kubectl get events --sort-by='.lastTimestamp'
```

输出类似：

```
LAST SEEN   TYPE      REASON              OBJECT              MESSAGE
2m          Normal    Scheduled           pod/my-pod          Successfully assigned default/my-pod to node-1
2m          Normal    Pulling             pod/my-pod          Pulling image "nginx:1.25"
90s         Normal    Pulled              pod/my-pod          Successfully pulled image "nginx:1.25"
90s         Normal    Created             pod/my-pod          Created container my-container
80s         Normal    Started             pod/my-pod          Started container my-container
10s         Warning   Unhealthy           pod/my-pod          Readiness probe failed: connection refused
```

### Event 的合并机制

Kubernetes 不会为每次相同事件创建新对象，而是合并重复事件：

- 第一次发生：创建 Event，Count=1
- 再次发生：更新同一个 Event，Count++，LastTimestamp 更新
- 这就是为什么你经常看到 `Warning BackOff ... 5 times` 这种输出

### Event 是一等资源

是的，Event 是标准的 Kubernetes API 资源（API Resource）。它有 TypeMeta + ObjectMeta，有自己的 REST endpoint：

```bash
# 这两种写法等价
kubectl get events
kubectl get events.v1.
```

Event 存储在 etcd 中，和普通 Pod、Service 一样，通过 API Server 暴露，支持 get/list/watch 操作。但它有一个特殊之处：有保留时间（默认1小时，kube-apiserver 的 `--event-ttl` 参数控制），过期后会被自动清理。

### 不同 Pod 的 Event 是独立的

每个 Event 实例严格绑定到一个 InvolvedObject（即特定的对象）。即使原因（Reason）和消息（Message）完全相同，只要涉及的对象不同，就是不同的 Event 实例。

**合并规则**：Event 的合并（去重）发生在同一个对象内部：

```
Pod-A 拉取镜像失败 → Event 1: {InvolvedObject: Pod-A, Reason: Failed, Count: 1}
Pod-A 再次拉取镜像失败 → Event 1 更新: {InvolvedObject: Pod-A, Reason: Failed, Count: 2}  ← 合并
Pod-B 拉取镜像失败 → Event 2: {InvolvedObject: Pod-B, Reason: Failed, Count: 1}             ← 新实例
```

Event 的 `metadata.name` 由事件记录器生成，格式通常是 `{involved-object-name}.{random-hex}`，比如 `my-pod-a.1764a3b2c...`，这保证了即使同一个 Pod 的多个不同类型 Event 也有各自独立的实例。

## 两者如何协作

以一个 Pod 启动失败的场景为例：

```
时间线:
T1  Scheduler → Event: "Successfully assigned pod to node-1"        (Normal)
T2  Kubelet   → Event: "Pulling image nginx:1.25"                   (Normal)
T3  Kubelet   → Event: "Started container"                          (Normal)
T4  Kubelet   → Event: "Readiness probe failed"                     (Warning)
T4  Kubelet   → Condition Ready=False, Reason=ContainersNotReady    ← 状态更新
T5  Kubelet   → Event: "Back-off restarting failed container"       (Warning, Count++)
T5  Kubelet   → Condition Ready=False 保持不变                       ← 没变化不更新时间
```

- Event 记录了每一步发生了什么（可审计、可排查）
- Condition 反映当前最终状态（可被其他控制器读取做决策）

## 其他资源中的 Condition

Condition 模式广泛存在于各种资源中：

| 资源 | 常见 Condition Type |
|------|---------------------|
| Node | Ready, MemoryPressure, DiskPressure, PIDPressure, NetworkUnavailable |
| Deployment | Available, Progressing, ReplicaFailure |
| Pod | Initialized, Ready, ContainersReady, PodScheduled |
| PVC | Bound |
| Job | Complete, Failed |

这些 Condition 让 `kubectl wait`、readiness gate、HPA 等上层机制能以统一的方式判断资源状态，而不需要理解每种资源的内部逻辑。

## Binding（绑定）

### 历史演变

Binding 曾经是 Kubernetes 的一种资源类型，但在 1.7 版本中已被废弃。

```go
// staging/src/k8s.io/api/core/v1/types.go:5714
// Binding ties one object to another; for example, a pod is bound to a node by a scheduler.
// Deprecated in 1.7, please use the bindings subresource of pods instead.
type Binding struct {
    metav1.TypeMeta   `json:",inline"`
    metav1.ObjectMeta `json:"metadata,omitempty"`
    Target            ObjectReference `json:"target"`  // 要绑定的目标对象
}
```

### 现在的用法：pods/binding 子资源

虽然 Binding 结构体仍然存在，但不再作为独立的 API 资源暴露。现在调度器通过 Pod 的子资源来完成绑定。

看调度器的实现（`pkg/scheduler/framework/plugins/defaultbinder/default_binder.go:51-62`）：

```go
func (b DefaultBinder) Bind(ctx context.Context, state *framework.CycleState,
    p *v1.Pod, nodeName string) *framework.Status {
    binding := &v1.Binding{
        ObjectMeta: metav1.ObjectMeta{Namespace: p.Namespace, Name: p.Name, UID: p.UID},
        Target:     v1.ObjectReference{Kind: "Node", Name: nodeName},
    }
    // 关键：调用的是 Pods().Bind() 子资源 API
    err := b.handle.ClientSet().CoreV1().Pods(binding.Namespace).Bind(ctx, binding, metav1.CreateOptions{})
    // ...
}
```

对应的 API 路径是：

```
POST /api/v1/namespaces/{namespace}/pods/{name}/binding
```

### 总结对比

| 项目 | 说明 |
|------|------|
| 过去 | Binding 是独立的资源类型，有自己的 REST endpoint `/api/v1/bindings` |
| 现在 | 通过 `pods/binding` 子资源实现，`POST /api/v1/namespaces/{ns}/pods/{name}/binding` |
| 数据结构 | `v1.Binding` 结构体仍在使用，但只是作为请求体传递 |
| 用途 | 调度器将 Pod 绑定到 Node 时使用 |
| 权限控制 | 子资源有独立的 RBAC 权限：`pods/binding` |

## 子资源与资源的区别

### 资源（Resource）

- **独立存储**：有自己的 etcd key，比如 `/registry/pods/{namespace}/{name}`
- **独立身份**：有自己的 `metadata.name`、`metadata.namespace`、`uid`
- **独立 REST endpoint**：`/api/v1/pods`
- **独立 RBAC 权限**：对 pods 资源的 get/list/watch/create/update/delete 单独授权

### 子资源（Subresource）

- **没有独立存储**：数据嵌在父对象中（比如 Pod 的 `.status` 字段）
- **没有独立身份**：依赖父对象的 metadata
- **有独立的 REST endpoint**：`/api/v1/pods/{name}/status`、`/api/v1/pods/{name}/log`
- **有独立的 RBAC 权限**：可以单独授权 `pods/status`、`pods/log` 等
- **有独立的操作语义**：比如 `pods/exec` 不修改对象，而是建立连接；`pods/binding` 只修改 `spec.nodeName`

### 具体例子

| 子资源 | Endpoint | 作用 | 修改什么 |
|--------|----------|------|----------|
| pods/status | PUT /pods/{name}/status | 更新 Pod 状态 | Pod 对象的 .status 字段 |
| pods/log | GET /pods/{name}/log | 获取容器日志 | 不修改对象 |
| pods/exec | POST /pods/{name}/exec | 在容器内执行命令 | 不修改对象 |
| pods/binding | POST /pods/{name}/binding | 将 Pod 绑定到 Node | 只修改 spec.nodeName |
| pods/scale | GET/PUT /pods/{name}/scale | 调整副本数 | 只修改 spec.replicas |

### 为什么需要子资源

**权限隔离**是最核心的原因。

比如：
- 应用开发者可以 update pods（改 spec），但不能改 pods/status
- kubelet 可以 update pods/status（报告状态），但不能改 pods 的 spec
- 如果 status 和 spec 是同一个资源的同一个字段，就没法做这种权限分离

子资源通过独立的 RBAC 规则实现了这种细粒度控制。

---

## 总结

| 概念 | 核心作用 | 关键特征 |
|------|----------|----------|
| Condition | 描述资源当前状态 | 三值逻辑、多维度、持久存在 |
| Event | 记录离散事件 | 时间戳、可合并、有保留时间 |
| Binding | Pod 绑定到 Node | 现在是 pods/binding 子资源 |
| 子资源 | 细粒度权限控制 | 独立 endpoint、共享存储 |

理解这些基础概念，是深入学习 Kubernetes Scheduler 的前提。下一篇文章，我们将从零开始了解 Scheduler 的整体架构。
