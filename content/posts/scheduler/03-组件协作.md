## 一、Scheduler ↔ API Server：唯一的通信对象

Scheduler 只和 API Server 交互，不直接接触 etcd、kubelet 或任何其他组件。交互方式有两种：

### 1. List & Watch（被动接收）

通过 client-go 的 Informer 机制，Scheduler 监听以下资源变化：

```go
// eventhandlers.go 中注册的监听
Pod (未调度)  → 加入调度队列
Pod (已调度)  → 更新 Cache 中的节点资源占用
Node          → 更新 Cache 中的节点信息
PV / PVC      → 触发等待卷的 Pod 重新调度
StorageClass  → 触发 WaitForFirstConsumer 的 Pod 重新调度
CSINode / CSIDriver / CSIStorageCapacity → 卷相关调度
```

**关键设计**：Pod Informer 注册了两套 Handler，通过 FilterFunc 区分：

```
Pod 事件
   ├── spec.nodeName 为空？ → addPodToSchedulingQueue()   // 进队列等待调度
   └── spec.nodeName 非空？ → addPodToCache()              // 记录已占用资源
```

### 2. Bind（主动写入）

调度完成后，Scheduler 调用 API Server 的 Bind 接口：

```go
// defaultbinder/default_binder.go:51
binding := &v1.Binding{
    ObjectMeta: metav1.ObjectMeta{Namespace: p.Namespace, Name: p.Name, UID: p.UID},
    Target:     v1.ObjectReference{Kind: "Node", Name: nodeName},
}
err := b.handle.ClientSet().CoreV1().Pods(binding.Namespace).Bind(ctx, binding, metav1.CreateOptions{})
```

这不是修改 Pod 对象，而是一个独立的 Binding 子资源。API Server 收到后：
1. 写入 etcd，设置 `pod.Spec.NodeName = nodeName`
2. 通知 kubelet

## 二、Scheduler ↔ kubelet：间接协作

Scheduler 和 kubelet 没有直接通信，完全通过 API Server 中转：

```
Scheduler                          API Server                         kubelet
    │                                  │                                 │
    │──── Bind(pod → node-A) ────────►│                                 │
    │                                  │──── Watch(pod updated) ────────►│
    │                                  │   pod.Spec.NodeName = "node-A"  │
    │                                  │                                 │
    │                                  │                      kubelet 发现
    │                                  │                      pod 绑到自己
    │                                  │                      ──► 拉镜像、建容器
    │                                  │                                 │
    │                                  │◄─── 更新 Pod Status ────────────│
    │                                  │    (Running, conditions...)     │
    │◄── Watch(pod status changed) ───│                                 │
    │    更新 Cache                    │                                 │
```

### 双重检查机制

有一个有趣的共享逻辑：Scheduler 和 kubelet 都用了同一套过滤逻辑来判断 Pod 能否放在某个节点上：

```go
// eventhandlers.go:504
// AdmissionCheck 被 kubelet (pkg/kubelet/lifecycle/predicate.go) 和 scheduler 共同使用
func AdmissionCheck(pod *v1.Pod, nodeInfo *framework.NodeInfo, ...) []AdmissionResult {
    // noderesources.Fits     - CPU/Memory 够不够
    // nodeaffinity           - 节点亲和性匹配
    // nodename               - 指定节点名
    // nodeports              - 端口冲突
}
```

这是双重检查：Scheduler 在调度时过滤一次，kubelet 在真正 admit 时再验证一次（因为调度决定和实际执行之间可能有时间差，节点资源可能已变化）。

## 三、Scheduler ↔ Scheduler（多副本）：Leader Election

生产环境通常部署多个 Scheduler 副本，但同一时刻只有一个在工作：

```go
// server.go:212-238
if cc.LeaderElection != nil {
    cc.LeaderElection.Callbacks = leaderelection.LeaderCallbacks{
        OnStartedLeading: func(ctx context.Context) {
            close(waitingForLeader)
            sched.Run(ctx)           // 获得锁 → 开始调度
        },
        OnStoppedLeading: func() {
            logger.Error(nil, "Leaderelection lost")
            klog.FlushAndExit(...)   // 丢失锁 → 进程退出
        },
    }
    leaderElector, err := leaderelection.NewLeaderElector(*cc.LeaderElection)
    leaderElector.Run(ctx)
}
```

通过 API Server 的 Lease 对象实现分布式锁。非 Leader 的实例待命，Leader 挂了立刻接管。

## 四、Scheduler 内部：Cache 与 Snapshot 的协作

这是理解 Scheduler 性能的关键：

```
  Informer 事件
      │
      ▼
 ┌──────────┐     UpdateSnapshot()     ┌──────────────┐
 │  Cache   │ ────────────────────────►│   Snapshot   │
 │(实时更新) │     每轮调度前拍快照      │ (本轮使用的) │
 └──────────┘                          └──────────────┘
      │                                      │
  含 assumed pods                        给 Filter/Score
  (乐观锁定的 Pod)                        插件使用
```

- **Cache**：实时更新，Informer 每来一个事件就更新。包含 assumed pod（调度完成但还没真正 bind 的 Pod）
- **Snapshot**：每轮 `schedulePod()` 开始时从 Cache 拍快照，保证本轮调度看到的是一致的节点状态

```go
// schedule_one.go:338
if err := sched.Cache.UpdateSnapshot(sched.nodeInfoSnapshot); err != nil {
    return result, err
}
```

### 为什么要 Snapshot？

如果直接读 Cache，在调度过程中（Filter/Score 并行执行时），Cache 可能被 Informer 事件更新，导致同一轮调度中不同节点看到的状态不一致。

Snapshot 机制保证了：
1. 一轮调度内看到的是同一个时间点的集群状态
2. 下一轮调度会拿到最新的 Snapshot
3. 避免了锁竞争和并发问题

## 五、集群事件 → 重新调度的触发链

Scheduler 不仅仅是"来了新 Pod 就调度"，很多集群事件会触发之前调度失败的 Pod 重新进入队列：

| 集群事件 | 触发的 Handler | 效果 |
|----------|----------------|------|
| Node 加入 | addNodeToCache | unschedulable 队列中的 Pod 移到 active 队列 |
| Node 资源变多 | updateNodeInCache | 同上（只在资源变"好"时触发） |
| Node 污点移除 | updateNodeInCache | 同上 |
| Pod 被删除 | deletePodFromCache | 释放资源，其他 Pod 有机会了 |
| PV 变为 Bound | PV Informer | 等待该 PV 的 Pod 重新调度 |
| StorageClass 创建 | onStorageClassAdd | WaitForFirstConsumer 的 PVC 可以绑了 |

这就是 `MoveAllToActiveOrBackoffQueue` 的作用 —— 把之前失败的 Pod 从 unschedulable 队列挪回 active 队列重试。

## 六、完整生命周期：一个 Pod 从创建到运行

```
1. 用户: kubectl apply → API Server 写入 etcd
                                    │
2. API Server:                      │
   Pod created (spec.nodeName="")   │
                                    │
3. Scheduler (Informer Watch):      │
   addPodToSchedulingQueue() ◄──────┘
   Pod 进入 PriorityQueue
                                    │
4. Scheduler (scheduleOne):         │
   取出 Pod → Filter → Score        │
   选出 node-3                      │
   AssumePod (缓存乐观锁定)         │
   Reserve → Permit                 │
                                    │
5. Scheduler (bindingCycle, goroutine):
   PreBind (如 Provision 卷)        │
   Bind → API Server ◄──────────────┘
          (创建 Binding 对象)
                                    │
6. API Server:                      │
   pod.spec.nodeName = "node-3"     │
   写入 etcd                        │
                                    │
7. kubelet (Watch):                 │
   发现新 Pod 绑到自己 ◄────────────┘
   拉镜像 → 创建容器 → 启动
   更新 pod.status = Running
                                    │
8. Scheduler (Informer Watch):      │
   addPodToCache() ◄────────────────┘
   Cache 中 assumed → confirmed
```

## 七、Scheduler 使用 Informer 的方式

Scheduler 需要缓存集群中所有 Pod、Node、CSI 等资源，用来做调度决策：

```go
// cmd/kube-scheduler/app/server.go:198-208
cc.InformerFactory.Start(ctx.Done())
cc.DynInformerFactory.Start(ctx.Done())
cc.InformerFactory.WaitForCacheSync(ctx.Done())

// pkg/scheduler/eventhandlers.go:251-389
func addAllEventHandlers(sched *Scheduler,
    informerFactory informers.SharedInformerFactory, ...) {

    // Pod 事件 → 更新调度器缓存 + 入队调度
    informerFactory.Core().V1().Pods().Informer().AddEventHandler(
        cache.FilteringResourceEventHandler{
            FilterFunc: assignedPod,          // 只关注已调度到节点的 Pod
            Handler: cache.ResourceEventHandlerFuncs{
                AddFunc:    sched.addPodToCache,
                UpdateFunc: sched.updatePodInCache,
                DeleteFunc: sched.deletePodFromCache,
            },
        },
    )

    // Node 事件 → 更新节点拓扑信息
    informerFactory.Core().V1().Nodes().Informer().AddEventHandler(
        cache.ResourceEventHandlerFuncs{
            AddFunc:    sched.addNodeToCache,
            UpdateFunc: sched.updateNodeInCache,
            DeleteFunc: sched.deleteNodeFromCache,
        },
    )

    // CSI Nodes/Driver/Capacity 事件
    informerFactory.Storage().V1().CSINodes().Informer().AddEventHandler(...)
    informerFactory.Storage().V1().CSIDrivers().Informer().AddEventHandler(...)

    // PV/PVC 事件
    informerFactory.Core().V1().PersistentVolumes().Informer().AddEventHandler(...)
    informerFactory.Core().V1().PersistentVolumeClaims().Informer().AddEventHandler(...)
}
```

Scheduler 的用法和 Controller 不太一样：它的 Handler 不是入 WorkQueue，而是直接更新内部缓存（SchedulingQueue + Snapshot），调度循环从 Snapshot 里读数据。

## 八、总结

Scheduler 是一个纯控制面组件，通过 Informer 感知集群状态，通过 Bind API 写入决策，和 kubelet 完全通过 API Server 解耦。

**核心设计要点**：

| 设计点 | 说明 |
|--------|------|
| 单一通信对象 | 只和 API Server 交互 |
| 间接协作 | 与 kubelet 通过 API Server 中转 |
| 高可用 | Leader Election 保证多副本 |
| 一致性 | Cache/Snapshot 机制保证调度决策 |
| 事件驱动 | 集群状态变化触发重新调度 |
