## 一句话概括

Informer 是 client-go 中的核心组件，它在本地维护一份与 API Server 最终一致的资源缓存，并在资源变化时以事件回调的方式通知使用者 —— 使得你无需频繁请求 API Server，也无需自己写轮询逻辑。

## 三大核心组件

源码中（`shared_informer.go:345-357`）明确描述了这三个组件：

### 1. Reflector — 反射器

```go
// reflector.go
// Reflector watches a specified resource and causes all changes to be
// reflected in the given store.
type Reflector struct {
    expectedType    reflect.Type
    store           Store          // 目标存储（DeltaFIFO）
    listerWatcher   ListerWatcher  // 封装 List/Watch API
    resyncPeriod    time.Duration
    ...
}
```

**职责**：对 API Server 执行 List（全量同步）和 Watch（增量监听），将变化推入 DeltaFIFO。

**启动流程**（`reflector.go:286-294`）：

```go
func (r *Reflector) Run(stopCh <-chan struct{}) {
    wait.BackoffUntil(func() {
        if err := r.ListAndWatch(stopCh); err != nil {
            r.watchErrorHandler(r, err)
        }
    }, r.backoffManager, true, stopCh)
}
```

- `ListAndWatch` 先执行一次 List（拉取全量），获得当前 resourceVersion
- 然后以该 resourceVersion 为起点进行 Watch（长连接，实时接收变更）
- 断线后自动带退避地重连（`BackoffUntil`）

### 2. DeltaFIFO — 增量队列

```go
// delta_fifo.go
// DeltaFIFO is a producer-consumer queue, where a Reflector is
// intended to be the producer, and the consumer is whatever calls
// the Pop() method.
//
// DeltaFIFO solves this use case:
//   - You want to process every object change (delta) at most once.
//   - When you process an object, you want to see everything
//     that's happened to it since you last processed it.
//   - You want to process the deletion of some of the objects.
```

**职责**：在 Reflector（生产者）和 HandleDeltas（消费者）之间做缓冲，保证：

| 特性 | 说明 |
|------|------|
| 合并变更 | 同一个 Key 的多次变更被合并为一个 Deltas 切片，避免重复处理 |
| 保留变更历史 | 消费时可以看到自上次处理以来该对象的所有变化 |
| 支持删除事件 | 通过与 KnownObjects（即 Indexer）配合，检测 Replace 时消失的对象并生成删除事件 |

**Delta 类型**有四种：`Added`、`Updated`、`Deleted`、`Replaced`（以及内部的 `Sync`）

### 3. Indexer — 带索引的本地缓存

```go
// index.go
// Indexer extends Store with multiple indices
type Indexer interface {
    Store
    Index(indexName string, obj interface{}) ([]interface{}, error)
    ByIndex(indexName, indexedValue string) ([]interface{}, error)
    AddIndexers(newIndexers Indexers) error
    ...
}
```

**职责**：
- 作为本地缓存，存储从 API Server 同步来的最新对象（每个 Key 只保留最新版本）
- 支持自定义索引，例如按 namespace、按 label 建立索引，让查询无需全量扫描

**Key 的格式**默认是 `namespace/name`（命名空间资源）或 `name`（集群资源），由 `MetaNamespaceKeyFunc` 决定。

## 事件分发：sharedProcessor

当 DeltaFIFO 中 Pop 出的 Deltas 被 `HandleDeltas` 处理时（`shared_informer.go:633`），每一步 Delta 会：
1. 更新 Indexer（本地缓存）
2. 触发 `OnAdd` / `OnUpdate` / `OnDelete`，调用 `processor.distribute()`

```go
func (s *sharedIndexInformer) OnAdd(obj interface{}, isInInitialList bool) {
    s.processor.distribute(addNotification{newObj: obj, isInInitialList: isInInitialList}, false)
}
```

`sharedProcessor` 管理一组 `processorListener`，每个 Listener 对应一个注册的 `ResourceEventHandler`：

```go
type sharedProcessor struct {
    listeners map[*processorListener]bool
    ...
}
```

多个 Handler 共享同一个缓存，这正是 "Shared" 的含义 —— 多个消费者共用同一个 Reflector 和 Indexer，避免对 API Server 造成重复压力。

## 完整启动流程

```go
// shared_informer.go:455-504
func (s *sharedIndexInformer) Run(stopCh <-chan struct{}) {
    // 1. 创建 DeltaFIFO，KnownObjects 指向 Indexer
    fifo := NewDeltaFIFOWithOptions(DeltaFIFOOptions{
        KnownObjects:          s.indexer,
        EmitDeltaTypeReplaced: true,
    })

    // 2. 创建 Controller（内部包含 Reflector）
    s.controller = New(cfg)  // cfg.Process = s.HandleDeltas

    // 3. 启动 Processor（事件分发协程）
    wg.StartWithChannel(processorStopCh, s.processor.run)

    // 4. 启动 Controller → Reflector 开始 ListAndWatch
    s.controller.Run(stopCh)
}
```

**启动顺序**（源码注释明确说明）：
1. 先启动 processor（消费者先就绪）
2. 再启动 controller（生产者开始 ListAndWatch）
3. 停止时：先停 controller，再停 processor（`processorStopCh` 的 defer 顺序保证）

## cache.Controller vs K8s Controller

这是 K8s 学习中的一个经典困惑点。它们不是同一个东西，但有密切关联。

源码 `controller.go` 开头第 31-37 行其实已经把关系说得很清楚了：

```go
// This file implements a low-level controller that is used in
// sharedIndexInformer, which is an implementation of
// SharedIndexInformer.  Such informers, in turn, are key components
// in the high level controllers that form the backbone of the
// Kubernetes control plane.
```

### 两种 "Controller" 的对比

| | cache.Controller（底层） | K8s Controller（高层） |
|---|---|---|
| 定义位置 | `client-go/tools/cache/controller.go` | `pkg/controller/replicaset/` 等各处 |
| 本质 | 一个数据管道：把 Reflector 和 DeltaFIFO 串起来的循环 | 一个业务控制器：调谐资源状态到期望状态 |
| 职责 | List+Watch → 入队 → Pop → 调用 ProcessFunc | 监听事件 → 计算期望与实际差异 → 创建/删除/更新资源 |
| 使用者 | 被 SharedInformer 内部使用，你不直接接触 | 你自己写的 / kube-controller-manager 里的各个控制器 |
| 典型例子 | `cache.New(cfg)` | ReplicaSetController, DeploymentController, NodeController… |

### 层次关系

```
┌─────────────────────────────────────────────────────────┐
│  K8s Controller（高层业务控制器）                          │
│  e.g. ReplicaSetController                              │
│                                                         │
│  ┌─ Informer（SharedIndexInformer）───────────────────┐  │
│  │                                                     │  │
│  │  ┌─ cache.Controller（底层数据管道）─────────────┐  │  │
│  │  │                                               │  │  │
│  │  │  Reflector ──► DeltaFIFO ──► Pop() ──► Process│  │  │
│  │  │      ▲                                         │  │  │
│  │  │      │ ListAndWatch                            │  │  │
│  │  └──────┼─────────────────────────────────────────┘  │  │
│  │         │                                            │  │
│  └─────────┼────────────────────────────────────────────┘  │
│            │                                               │
│         API Server                                         │
│                                                            │
│  ┌─ WorkQueue ◄────── Informer EventHandler 入队 ──────┐  │
│  │                                                      │  │
│  │  Worker 协程消费 ──► syncReplicaSet() 业务逻辑       │  │
│  └──────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

## 实际使用方式

### 基本用法

```go
// 1. 创建 Informer（通常通过 SharedInformerFactory）
factory := informers.NewSharedInformerFactory(clientset, 30*time.Second)
podInformer := factory.Core().V1().Pods().Informer()

// 2. 注册事件处理器
podInformer.AddEventHandler(cache.ResourceEventHandlerFuncs{
    AddFunc:    func(obj interface{}) { /* 处理新增 */ },
    UpdateFunc: func(oldObj, newObj interface{}) { /* 处理更新 */ },
    DeleteFunc: func(obj interface{}) { /* 处理删除 */ },
})

// 3. 启动
stopCh := make(chan struct{})
factory.Start(stopCh)
factory.WaitForCacheSync(stopCh)  // 等待首次全量同步完成
```

### 配合 WorkQueue 使用（推荐）

源码注释（`shared_informer.go:126-130`）明确指出：

> A client must process each notification promptly; a SharedInformer is not
> engineered to deal well with a large backlog of notifications. Lengthy
> processing should be passed off to something else, for example through a
> client-go/util/workqueue.

```go
queue := workqueue.NewRateLimitingQueue(...)

podInformer.AddEventHandler(cache.ResourceEventHandlerFuncs{
    AddFunc: func(obj interface{}) {
        key, _ := cache.MetaNamespaceKeyFunc(obj)
        queue.Add(key)  // 快速入队，不在回调里做重活
    },
    // ...
})

// Worker 协程异步消费
go func() {
    for processNextItem(queue) { }
}()
```

## SharedInformer vs 普通 Informer

| 对比项 | 普通 Informer | SharedInformer |
|--------|---------------|----------------|
| API Server 压力 | 每个消费者各建一个 Watch | 所有消费者共享一个 Watch |
| 内存占用 | 每个消费者各有一份缓存 | 共享一份 Indexer 缓存 |
| 使用方式 | 已不推荐直接使用 | 通过 SharedInformerFactory 获取 |

## 关键设计要点总结

| 设计点 | 体现 |
|--------|------|
| 最终一致性 | 缓存不保证实时，但保证"最终"与 API Server 一致（源码注释第一段） |
| 事件有序性 | 同一对象的事件按序到达，但不同对象之间无顺序保证 |
| UID 不参与身份识别 | 对象身份由 namespace/name 决定，若对象被删除后重建同名，可能只收到一次 Update |
| Resync 机制 | 定期将缓存中的对象以 Update 事件重新发给 Handler，用于处理"错过的事件"或状态对账 |
| TransformFunc | 可在存入缓存前转换对象，常见用途：裁剪无用字段以节省内存 |
