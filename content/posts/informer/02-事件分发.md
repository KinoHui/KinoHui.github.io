## 完整数据流：谁消费谁，怎么配合

### 先纠正一个常见误解

Indexer 不是 DeltaFIFO 的消费者。准确地说是 `HandleDeltas` 这个函数在消费，它同时做两件事：更新 Indexer + 分发事件给 sharedProcessor。

看 `controller.go:190-203`，消费 DeltaFIFO 的是一个死循环 `processLoop`：

```go
func (c *controller) processLoop() {
    for {
        obj, err := c.config.Queue.Pop(PopProcessFunc(c.config.Process))
        //                                  ↑
        //                    c.config.Process = s.HandleDeltas
    }
}
```

而 `HandleDeltas` 被调用后，最终走到 `processDeltas`（`controller.go:440-472`）：

```go
func processDeltas(handler ResourceEventHandler, clientState Store,
                   deltas Deltas, isInInitialList bool) error {
    for _, d := range deltas {
        obj := d.Object
        switch d.Type {
        case Sync, Replaced, Added, Updated:
            if old, exists, _ := clientState.Get(obj); exists {
                clientState.Update(obj)   // ① 更新 Indexer
                handler.OnUpdate(old, obj)// ② 通知 sharedProcessor
            } else {
                clientState.Add(obj)      // ① 写入 Indexer
                handler.OnAdd(obj, ...)   // ② 通知 sharedProcessor
            }
        case Deleted:
            clientState.Delete(obj)       // ① 从 Indexer 删除
            handler.OnDelete(obj)         // ② 通知 sharedProcessor
        }
    }
}
```

其中 `handler` 就是 `sharedIndexInformer`，`clientState` 就是 `indexer`。

## Indexer 内部是怎么工作的

Indexer 的实现是 `cache` 结构体（`store.go:139-145`），内部包了一个 `threadSafeMap`：

```go
type cache struct {
    cacheStorage ThreadSafeStore  // 真正的存储
    keyFunc      KeyFunc          // 对象 → Key 的转换函数
}
```

`threadSafeMap` 的实际结构（`thread_safe_store.go:209-215`）：

```go
type threadSafeMap struct {
    lock  sync.RWMutex
    items map[string]interface{}  // 主存储：key → 对象
    index *storeIndex             // 索引结构
}
```

`storeIndex` 的数据结构（`thread_safe_store.go:63-68`）：

```go
type storeIndex struct {
    indexers Indexers  // map[索引名]IndexFunc   索引名 → 索引函数
    indices  Indices   // map[索引名]Index        索引名 → 倒排索引
}
```

其中 `Index` 是（`index.go:95`）：

```go
type Index map[string]sets.String  // 索引值 → 存储Key的集合
```

### 用一个具体例子来说明

假设你对 Pod Informer 注册了一个 `NamespaceIndex`：

```go
informer.AddIndexers(cache.Indexers{
    cache.NamespaceIndex: cache.MetaNamespaceIndexFunc,
})
```

内存中的结构大概长这样：

```
items (主存储)
┌───────────────────────────────────────────┐
│ "default/nginx-abc"   → Pod{...}         │
│ "default/redis-xyz"   → Pod{...}         │
│ "kube-system/coredns" → Pod{...}         │
└───────────────────────────────────────────┘

indices["namespace"] (倒排索引)
┌───────────────────────────────────────────┐
│ "default"      → {"default/nginx-abc",    │
│                    "default/redis-xyz"}   │
│ "kube-system"  → {"kube-system/coredns"}  │
└───────────────────────────────────────────┘
```

### 写入时的索引维护

当调用 `Update` 时（`thread_safe_store.go:221-227`）：

```go
func (c *threadSafeMap) Update(key string, obj interface{}) {
    c.lock.Lock()
    defer c.lock.Unlock()
    oldObject := c.items[key]
    c.items[key] = obj
    c.index.updateIndices(oldObject, obj, key)  // 同步更新索引
}
```

`updateIndices`（`thread_safe_store.go:143-183`）的核心逻辑：

```go
func (i *storeIndex) updateIndices(oldObj, newObj interface{}, key string) {
    for name, indexFunc := range i.indexers {
        // 1. 用 IndexFunc 算出旧对象和新对象的索引值
        oldIndexValues, _ := indexFunc(oldObj)
        indexValues, _    := indexFunc(newObj)

        // 2. 如果索引值没变（最常见情况），跳过——性能优化
        if len(indexValues) == 1 && len(oldIndexValues) == 1
           && indexValues[0] == oldIndexValues[0] {
            continue
        }

        // 3. 从旧索引值中删除这个 key
        for _, value := range oldIndexValues {
            i.deleteKeyFromIndex(key, value, index)
        }
        // 4. 在新索引值中添加这个 key
        for _, value := range indexValues {
            i.addKeyToIndex(key, value, index)
        }
    }
}
```

### 查询时怎么用

当你写 Controller 时，用的是 Lister 封装：

```go
// 列出 default 命名空间下所有 Pod —— 走索引，O(1) 查倒排表

```

## sharedProcessor 是怎么工作的

### 核心结构

```go
type sharedProcessor struct {
    listeners map[*processorListener]bool  // 所有已注册的 Handler
}
```

### 事件分发：distribute()

`shared_informer.go:774-790`：

```go
func (p *sharedProcessor) distribute(obj interface{}, sync bool) {
    p.listenersLock.RLock()
    defer p.listenersLock.RUnlock()

    for listener, isSyncing := range p.listeners {
        switch {
        case !sync:
            listener.add(obj)    // 非Sync事件：发给所有 Listener
        case isSyncing:
            listener.add(obj)    // Sync事件：只发给需要resync的 Listener
        default:
            // 不需要resync的 Listener 跳过 Sync 事件
        }
    }
}
```

这里的 **sync 事件** 是指 ResourceVersion 未变化的 Update（即 Resync 产生的），只在需要定期全量对账的 Handler 才收到。

### 每个 processorListener 的内部结构

`shared_informer.go:864-899`，每个 Listener 有两个协程 + 两个 channel + 一个无界环形缓冲区：

```
sharedProcessor.distribute()
        │
        │ listener.add(notification)
        ▼
      addCh (无缓冲 channel)
        │
        │ ◄──── pop() 协程搬运
        │
        ▼
  pendingNotifications (无界环形缓冲区)
        │
        │ ◄──── pop() 协程搬运
        ▼
      nextCh (无缓冲 channel)
        │
        │ ◄──── run() 协程消费
        ▼
  handler.OnAdd / OnUpdate / OnDelete
```

### pop() 协程

`shared_informer.go:931-959`：

```go
func (p *processorListener) pop() {
    defer close(p.nextCh)

    var nextCh chan<- interface{}
    var notification interface{}
    for {
        select {
        case nextCh <- notification:          // 往 nextCh 发
            notification, ok = p.pendingNotifications.ReadOne()
            if !ok {
                nextCh = nil                  // 缓冲区空了，关闭这个 case
            }
        case notificationToAdd, ok := <-p.addCh: // 从 addCh 收
            if !ok { return }
            if notification == nil {
                notification = notificationToAdd   // 快速路径：直接转发
                nextCh = p.nextCh
            } else {
                p.pendingNotifications.WriteOne(notificationToAdd) // 慢路径：入缓冲区
            }
        }
    }
}
```

### run() 协程

`shared_informer.go:961-986`：

```go
func (p *processorListener) run() {
    stopCh := make(chan struct{})
    wait.Until(func() {
        for next := range p.nextCh {
            switch notification := next.(type) {
            case updateNotification:
                p.handler.OnUpdate(notification.oldObj, notification.newObj)
            case addNotification:
                p.handler.OnAdd(notification.newObj, notification.isInInitialList)
            case deleteNotification:
                p.handler.OnDelete(notification.oldObj)
            }
        }
        close(stopCh)
    }, 1*time.Second, stopCh)
}
```

### 为什么设计成两个协程 + 缓冲区？

```
┌──────────────────────────────────────────────────────────┐
│              为什么要这个 pop() + RingGrowing？             │
│                                                          │
│  distribute() 是在 HandleDeltas 里同步调用的，              │
│  而 HandleDeltas 持有 DeltaFIFO 的锁。                    │
│                                                          │
│  如果 distribute() 直接等你的 Handler 执行完，              │
│  你的 Handler 如果慢了，就会阻塞 DeltaFIFO 的 Pop，         │
│  导致整个 Informer 卡住。                                  │
│                                                          │
│  所以：                                                    │
│  addCh ← add() 是阻塞发送（背压）                          │
│  pop() 负责从 addCh 搬到 nextCh，中间用 RingGrowing 缓冲    │
│  run() 独立执行 Handler，慢了只会撑大缓冲区，不会阻塞 FIFO   │
│                                                          │
│  代价：一个慢 Handler 会无限撑大内存（源码注释也承认了）      │
│  "a failing/stalled listener will have infinite            │
│   pendingNotifications added until we OOM"                 │
└──────────────────────────────────────────────────────────┘
```

## 多 Controller 共享 Informer

### 真实场景：Pod Informer 被 13 个 Controller 共享

在 kube-controller-manager 中，以下 Controller 全都对同一个 Pod Informer 注册了 Handler：

```
controllerContext.InformerFactory   ← 所有 Controller 共享这一个 Factory
    │
    ├── startReplicaSetController  → .Core().V1().Pods().Informer().AddEventHandler(...)
    ├── startDeploymentController  → .Core().V1().Pods().Informer().AddEventHandler(...)
    ├── startJobController         → .Core().V1().Pods().Informer().AddEventHandler(...)
    ├── startDaemonSetController   → .Core().V1().Pods().Informer().AddEventHandler(...)
    ├── startStatefulSetController → .Core().V1().Pods().Informer().AddEventHandler(...)
    ├── startEndpointController    → .Core().V1().Pods().Informer().AddEventHandler(...)
    ├── startEndpointSliceCtrl     → .Core().V1().Pods().Informer().AddEventHandler(...)
    ├── startDisruptionController  → .Core().V1().Pods().Informer().AddEventHandler(...)
    ├── startNodeLifecycleCtrl     → .Core().V1().Pods().Informer().AddEventHandler(...)
    ├── startAttachDetachCtrl      → .Core().V1().Pods().Informer().AddEventHandler(...)
    ├── startPVCProtectionCtrl     → .Core().V1().Pods().Informer().AddEventHandler(...)
    ├── startEphemeralVolumeCtrl   → .Core().V1().Pods().Informer().AddEventHandler(...)
    └── startResourceClaimCtrl     → .Core().V1().Pods().Informer().AddEventHandler(...)
```

每个 Controller 的 Handler 做的事情完全不同：

| Controller | Handler 做了什么 |
|------------|------------------|
| ReplicaSetController | AddFunc: rsc.addPod → 找到 Pod 的 ownerRef → 把对应的 ReplicaSet 入队 |
| DeploymentController | 只注册了 DeleteFunc: dc.deletePod → 找到 ownerRef → 把对应的 Deployment 入队 |
| JobController | AddFunc: jm.addPod → 找到 ownerRef → 把对应的 Job 入队 |
| DaemonSetController | AddFunc: dsc.addPod → 找到 ownerRef → 把对应的 DaemonSet 入队 |
| ... | 各自关心各自的 ownerRef |

### 为什么 13 个 Controller 拿到的是同一个 Informer？

关键在于 `InformerFor` 的去重机制（`factory.go:189-208`）：

```go
func (f *sharedInformerFactory) InformerFor(obj runtime.Object,
    newFunc internalinterfaces.NewInformerFunc) cache.SharedIndexInformer {

    informerType := reflect.TypeOf(obj)  // *v1.Pod — 每次都是同一个类型

    informer, exists := f.informers[informerType]
    if exists {
        return informer    // ★ 已存在，直接返回同一个实例！
    }

    // 第一次调用才创建
    informer = newFunc(f.client, resyncPeriod)
    f.informers[informerType] = informer
    return informer
}
```

而 `Pods()` 每次返回的是一个轻量的 `podInformer` 包装（`core/v1/interface.go:118`）：

```go
func (v *version) Pods() PodInformer {
    return &podInformer{factory: v.factory, ...}  // 只是包装，不是真正的 Informer
}
```

只有调 `.Informer()` 时才触发 `InformerFor`，而 `InformerFor` 用 `reflect.TypeOf(&v1.Pod{})` 做 key，所以无论调多少次，返回的都是同一个 `sharedIndexInformer` 实例。

### 13 个 Handler 注册后，内存里长什么样

```
sharedIndexInformer (Pod)          ← 整个 kube-controller-manager 中只有一个
│
├── indexer (Indexer)              ← 只有一份 Pod 缓存
│
├── Reflector                      ← 只有一个 Watch 连接
│   └── DeltaFIFO                  ← 只有一个队列
│
├── processLoop                    ← 只有一个消费循环
│   └── HandleDeltas → processDeltas
│         ├── indexer.Add/Update/Delete  (更新唯一缓存)
│         └── self.OnAdd/OnUpdate/OnDelete
│               └── processor.distribute(notification)
│                     │
│                     ▼
│              sharedProcessor
│              ┌──────────────────────────────────────────────┐
│              │ listeners:                                    │
│              │   listener_0 (ReplicaSet handler) → true      │
│              │   listener_1 (Deployment handler) → true      │
│              │   listener_2 (Job handler)        → true      │
│              │   listener_3 (DaemonSet handler)  → true      │
│              │   ...                                         │
│              └──────────────────────────────────────────────┘
```

每个 listener 都有自己独立的：
- `addCh` channel
- `nextCh` channel
- `pendingNotifications` ring buffer
- `pop()` goroutine
- `run()` goroutine
- 指向各自 Controller 的 handler

## 一个 Pod 事件如何扇出到 13 个 Handler

假设 API Server 推送了一个 Pod Added 事件：

```
Reflector (唯一的 Watch 连接)
  │
  │ watch.Added → DeltaFIFO.Add(Pod{default/nginx})
  │
  ▼
DeltaFIFO
  │
  │ processLoop: Pop() → HandleDeltas()
  │
  ▼
processDeltas(self, indexer, deltas)
  │
  ├── ① indexer.Add(Pod)         ← 唯一的缓存被更新
  │
  └── ② self.OnAdd(Pod)          ← sharedIndexInformer.OnAdd
          │
          ▼
      processor.distribute(addNotification{Pod}, sync=false)
          │
          │  for listener, _ := range p.listeners {
          │      listener.add(obj)     ← 遍历所有 13 个 listener
          │  }
          │
          ├──► listener_0.addCh  ←──── ReplicaSet 的 pop() 搬运 → run() → rsc.addPod()
          │                                   │                         找到 ownerRef
          │                                   │                         → RS workqueue
          │                                   ▼
          │
          ├──► listener_1.addCh  ←──── Deployment 的 pop() → run() → dc.deletePod()
          │                                   │
          │                            (只注册了 Delete)
          │                                   ▼
          │
          ├──► listener_2.addCh  ←──── Job 的 pop() → run() → jm.addPod()
          │                                   │                     找到 ownerRef
          │                                   │                     → Job workqueue
          │                                   ▼
          │
          ├──► listener_3.addCh  ←──── DaemonSet 的 pop() → run() → dsc.addPod()
          │
          ├──► ... (listener_4 ~ listener_12 同理)
          │
          └──► listener_12.addCh ←──── ResourceClaim 的 pop() → run() → handler()
```

### 13 个 listener 是并发执行 Handler 的

每个 listener 的 `run()` 是独立的 goroutine，所以 13 个 Handler 并发执行：

```
                    distribute() 在持有 blockDeltas 锁的情况下
                    顺序把通知塞入 13 个 listener 的 addCh
                    ┌────────────────────────────────────┐
                    │  for listener := range listeners { │
                    │      listener.add(notification)    │  ← 顺序执行
                    │  }                                 │
                    └────────────────────────────────────┘
                                    │
          ┌─────────────────────────┼─────────────────────────┐
          │                         │                         │
    listener_0               listener_1               listener_2
    pop() → nextCh          pop() → nextCh          pop() → nextCh
       │                       │                       │
    run() (goroutine)       run() (goroutine)       run() (goroutine)
       │                       │                       │
    rsc.addPod()            dc.deletePod()          jm.addPod()
       │                       │                       │
    RS workqueue.Add()      Deploy workqueue.Add()  Job workqueue.Add()

          └──────── 三者并发执行，互不干扰 ────────────┘
```

## kube-controller-manager 是一个进程

是的，`kube-controller-manager` 是单个进程：

```go
// cmd/kube-controller-manager/controller-manager.go:34
func main() {
    command := app.NewControllerManagerCommand()
    code := cli.Run(command)
    os.Exit(code)
}
```

一个 `main()`，一个二进制，一个进程。所有 Controller 都在这个进程里以 goroutine 运行。

### 共享一个 InformerFactory

`controllermanager.go:519-557` — 创建 ControllerContext 时只创建了一个 Factory：

```go
func CreateControllerContext(...) (ControllerContext, error) {
    // ★ 整个进程只创建一个 Factory
    sharedInformers := informers.NewSharedInformerFactory(versionedClient, ResyncPeriod(s)())

    ctx := ControllerContext{
        InformerFactory: sharedInformers,   // ★ 所有 Controller 共享这一个 Factory
        ...
    }
}
```

### 完整的调用链

```
main()                                          ← 一个进程
  └── Run()
        └── sharedInformers = NewSharedInformerFactory(...)     ← 一个 Factory
        │
        └── ControllerContext{InformerFactory: sharedInformers} ← 传同一个引用
        │
        └── StartControllers()
              │
              ├── startReplicaSetController(ctx, controllerCtx)
              │     controllerCtx.InformerFactory.Core().V1().Pods()
              │     └── InformerFor(*Pod) → informers[*Pod] 命中！返回已有实例
              │     └── .AddEventHandler(rsc.addPod)           ← listener_0
              │     └── go rsc.Run()                           ← goroutine
              │
              ├── startDeploymentController(ctx, controllerCtx)
              │     controllerCtx.InformerFactory.Core().V1().Pods()
              │     └── InformerFor(*Pod) → informers[*Pod] 命中！返回同一个实例
              │     └── .AddEventHandler(dc.deletePod)         ← listener_1
              │     └── go dc.Run()                            ← goroutine
              │
              ├── ... (共注册了 ~40 个 Controller)
              │
              └── controllerCtx.InformerFactory.Start(stopCh)   ← 统一启动
```

## 不同进程之间不能共享

SharedInformer 的"共享"是进程内内存级别的共享 —— 多个 Controller 持有同一个 Go 对象的指针。

不同进程之间：

```
  kube-controller-manager (进程 A)          cloud-controller-manager (进程 B)
  ┌──────────────────────────┐             ┌──────────────────────────┐
  │ SharedInformerFactory    │             │ SharedInformerFactory    │
  │ ┌──────────────────────┐ │             │ ┌──────────────────────┐ │
  │ │ Pod Informer         │ │             │ │ Node Informer        │ │
  │ │  Reflector ──Watch──►│─┼──API Server─┼─│  Reflector ──Watch──►│ │
  │ │  DeltaFIFO           │ │             │ │  DeltaFIFO           │ │
  │ │  Indexer (缓存副本A) │ │             │ │  Indexer (缓存副本B) │ │
  │ └──────────────────────┘ │             │ └──────────────────────┘ │
  └──────────────────────────┘             └──────────────────────────┘

  各自独立的：
    - Watch 连接
    - DeltaFIFO 队列
    - Indexer 缓存
    - sharedProcessor + listener
```

每个进程都独立：
1. 各自建 Watch 连接 → API Server 看到的连接数是 **进程数 × 资源类型数**
2. 各自维护缓存 → 内存占用是重复的
3. 各自消费事件 → API Server 会给每个 Watch 连接都推送一份事件

这也是为什么 K8s 把大多数 Controller 合并到一个进程里运行的核心原因 —— 减少 Watch 连接数、减少内存、减少 API Server 的压力。
