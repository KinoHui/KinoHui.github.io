## Informer 全生命周期：从一个 Pod 被创建到你的 Handler 被调用

### 用户代码（以 sample-controller 为原型）

```go
// ① 创建 Factory
kubeInformerFactory := kubeinformers.NewSharedInformerFactory(kubeClient, time.Second*30)

// ② 获取 PodInformer（此时 Informer 被惰性创建，但还没启动）
podInformer := kubeInformerFactory.Core().V1().Pods()

// ③ 注册事件 Handler
podInformer.Informer().AddEventHandler(cache.ResourceEventHandlerFuncs{
    AddFunc:    func(obj interface{}) { fmt.Println("Pod added!") },
    UpdateFunc: func(old, new interface{}) { fmt.Println("Pod updated!") },
    DeleteFunc: func(obj interface{}) { fmt.Println("Pod deleted!") },
})

// ④ 启动所有 Informer
stopCh := make(chan struct{})
kubeInformerFactory.Start(stopCh)

// ⑤ 等待缓存同步
cache.WaitForCacheSync(stopCh, podInformer.Informer().HasSynced)
```

下面追踪一个 Pod（假设叫 `default/nginx`）从 API Server 到你 Handler 的完整旅程。

## 第一阶段：Informer 是怎么被创建出来的

### 1. 调用 podInformer.Informer() — 惰性创建

```go
// informers/core/v1/pod.go:84
func (f *podInformer) Informer() cache.SharedIndexInformer {
    return f.factory.InformerFor(&corev1.Pod{}, f.defaultInformer)
}
```

进入 `InformerFor`（`factory.go:189-208`）：

```go
func (f *sharedInformerFactory) InformerFor(obj runtime.Object,
    newFunc internalinterfaces.NewInformerFunc) cache.SharedIndexInformer {

    informerType := reflect.TypeOf(obj)  // *v1.Pod
    informer, exists := f.informers[informerType]
    if exists { return informer }  // 已存在则复用

    // 创建 Pod Informer
    informer = newFunc(f.client, resyncPeriod)
    f.informers[informerType] = informer
    return informer
}
```

`newFunc` 就是 `defaultInformer`（`pod.go:80-82`），它最终调用：

```go
// pod.go:58-78
func NewFilteredPodInformer(...) cache.SharedIndexInformer {
    return cache.NewSharedIndexInformer(
        &cache.ListWatch{
            // 这两个函数封装了对 API Server 的请求
            ListFunc: func(options metav1.ListOptions) (runtime.Object, error) {
                return client.CoreV1().Pods(namespace).List(ctx, options)
            },
            WatchFunc: func(options metav1.ListOptions) (watch.Interface, error) {
                return client.CoreV1().Pods(namespace).Watch(ctx, options)
            },
        },
        &corev1.Pod{},
        resyncPeriod,
        indexers,
    )
}
```

这一步创建了 `sharedIndexInformer` 结构体，内部有：
- `indexer`：本地缓存（Indexer），此时为空
- `processor`：事件分发器（sharedProcessor），此时为空
- `listerWatcher`：封装了 List/Watch 的函数对象

**注意**：此时没有任何 goroutine 在跑，Reflector 还没出生。

### 2. 注册 Handler — AddEventHandler

```go
podInformer.Informer().AddEventHandler(cache.ResourceEventHandlerFuncs{...})
```

进入 `shared_informer.go:555-604`：

```go
func (s *sharedIndexInformer) AddEventHandler(handler ResourceEventHandler) {

    // 创建一个 processorListener 包装你的 handler

    //                ├── handler: 你的 ResourceEventHandlerFuncs
    //                └── pendingNotifications: RingGrowing (无界环形缓冲区)

```

进入 `sharedProcessor.addListener`（`shared_informer.go:732-748`）：

```go
func (p *sharedProcessor) addListener(listener *processorListener) {
    if p.listeners == nil {
        p.listeners = make(map[*processorListener]bool)
    }
    p.listeners[listener] = true   // 加入 map，value=true 表示"正在 syncing"

    // 因为还没 start，这里不会启动 listener 的 goroutine

```

此时状态：processor 的 listeners map 里有一个 listener，但它的 `pop()` 和 `run()` goroutine 都还没启动。

## 第二阶段：启动 — factory.Start()

```go
kubeInformerFactory.Start(stopCh)
```

进入 `factory.go:133-155`：

```go
func (f *sharedInformerFactory) Start(stopCh <-chan struct{}) {
    for informerType, informer := range f.informers {
        if !f.startedInformers[informerType] {
            go informer.Run(stopCh)   // 为每个 informer 起义个 goroutine
            f.startedInformers[informerType] = true
        }
    }
}
```

进入 `sharedIndexInformer.Run()`（`shared_informer.go:455-504`），这是整个引擎的点火时刻：

```go
func (s *sharedIndexInformer) Run(stopCh <-chan struct{}) {
    // ── Step 1: 创建 DeltaFIFO ──
    fifo := NewDeltaFIFOWithOptions(DeltaFIFOOptions{
        KnownObjects:          s.indexer,      // 指向本地缓存
        EmitDeltaTypeReplaced: true,
    })

    // ── Step 2: 创建 Controller（内嵌 Reflector）──
    cfg := &Config{
        Queue:            fifo,
        ListerWatcher:    s.listerWatcher,    // List/Watch 函数
        Process:          s.HandleDeltas,      // Pop 出来后调这个
        FullResyncPeriod: s.resyncCheckPeriod,
        ShouldResync:     s.processor.shouldResync,
    }
    s.controller = New(cfg)

    // ── Step 3: 启动 processor（消费者先就绪）──
    wg.StartWithChannel(processorStopCh, s.processor.run)

    // ── Step 4: 启动 controller（生产者开始工作）──
    s.controller.Run(stopCh)
}
```

### 3. processor.run() — 启动所有 listener 的 goroutine

```go
// shared_informer.go:792-818
func (p *sharedProcessor) run(stopCh <-chan struct{}) {
    for listener := range p.listeners {
        p.wg.Start(listener.run)   // 启动 run goroutine
        p.wg.Start(listener.pop)   // 启动 pop goroutine
    }
    p.listenersStarted = true
    <-stopCh
    // stopCh 关闭后，关闭所有 listener 的 addCh...
}
```

此时你的 listener 的 `pop()` 和 `run()` 两个 goroutine 已经在阻塞等待了。

### 4. controller.Run() — 启动 Reflector + processLoop

```go
// controller.go:130-165
func (c *controller) Run(stopCh <-chan struct{}) {
    // 创建 Reflector
    r := NewReflectorWithOptions(c.config.ListerWatcher, ...)

    // 启动 Reflector（生产者）
    wg.StartWithChannel(stopCh, r.Run)

    // 启动 processLoop（消费者循环）
    wait.Until(c.processLoop, time.Second, stopCh)
}
```

此时两个关键 goroutine 都启动了：
- `Reflector.Run` — 开始 List+Watch
- `processLoop` — 开始循环 Pop DeltaFIFO

## 第三阶段：Reflector 首次 List — 全量拉取

### 5. Reflector.Run() → ListAndWatch() → list()


`list()` 调用你之前传入的 `ListFunc`，即向 API Server 发 HTTP 请求：

```
GET /api/v1/pods
```

API Server 返回一个 PodList，假设里面有 3 个 Pod：

```go

```

### 6. syncWith() → DeltaFIFO.Replace()

```go
// reflector.go:681-687
func (r *Reflector) syncWith(items []runtime.Object, resourceVersion string) error {
    found := make([]interface{}, 0, len(items))
    for _, item := range items {
        found = append(found, item)
    }
    return r.store.Replace(found, resourceVersion)  // store 就是 DeltaFIFO
}
```

`DeltaFIFO.Replace()`（`delta_fifo.go:623-632`）为每个 Pod 创建一个 `Replaced` 类型的 Delta：

```
DeltaFIFO 内部状态（List 后）：
┌──────────────────────────────────────────────────────────────┐
│ queue: ["default/nginx", "default/redis", "kube-system/coredns"]   │
│                                                              │
│ items:                                                       │
│   "default/nginx"       → Deltas[{Type:Replaced, Pod{...}}]  │
│   "default/redis"       → Deltas[{Type:Replaced, Pod{...}}]  │
│   "kube-system/coredns" → Deltas[{Type:Replaced, Pod{...}}]  │
└──────────────────────────────────────────────────────────────┘
```

同时 `cond.Broadcast()` 唤醒正在 Pop 上阻塞的 `processLoop`。

## 第四阶段：processLoop 消费 — Pop → HandleDeltas

### 7. processLoop 循环 Pop

```go
// controller.go:190-203
func (c *controller) processLoop() {
    for {
        obj, err := c.config.Queue.Pop(PopProcessFunc(c.config.Process))
        //                                  ↑                    ↑
        //                              DeltaFIFO           HandleDeltas
    }
}
```

`DeltaFIFO.Pop()`（`delta_fifo.go:563-612`）从队列头部取出一个 item，然后在 Pop 内部直接调用 process：

```go
func (f *DeltaFIFO) Pop(process PopProcessFunc) (interface{}, error) {
    f.lock.Lock()
    defer f.lock.Unlock()
    for {
        // 队列为空时阻塞等待
        for len(f.queue) == 0 {
            f.cond.Wait()
        }
        id := f.queue[0]           // "default/nginx"
        f.queue = f.queue[1:]      // 出队
        item := f.items[id]        // Deltas[{Replaced, Pod}]
        delete(f.items, id)

        // ★ 关键：在持有 FIFO 锁的情况下调用 process
        err := process(item, isInInitialList)
        return item, err
    }
}
```

### 8. HandleDeltas → processDeltas

`process` 就是 `HandleDeltas`（`shared_informer.go:633-641`）：

```go
func (s *sharedIndexInformer) HandleDeltas(obj interface{}, isInInitialList bool) error {
    s.blockDeltas.Lock()
    defer s.blockDeltas.Unlock()

    deltas := obj.(Deltas)
    return processDeltas(s, s.indexer, deltas, isInInitialList)
    //              ↑        ↑
    //          handler    clientState(Indexer)
}
```

`processDeltas`（`controller.go:440-472`）是双写的核心逻辑：

```go
func processDeltas(handler ResourceEventHandler, clientState Store,
                   deltas Deltas, isInInitialList bool) error {
    for _, d := range deltas {
        obj := d.Object  // Pod{default/nginx}

        switch d.Type {
        case Sync, Replaced, Added, Updated:
            if old, exists, _ := clientState.Get(obj); exists {
                clientState.Update(obj)         // ① 更新 Indexer
                handler.OnUpdate(old, obj)      // ② 通知 sharedProcessor
            } else {
                clientState.Add(obj)            // ① 写入 Indexer
                handler.OnAdd(obj, isInInitialList) // ② 通知 sharedProcessor
            }
        case Deleted:
            clientState.Delete(obj)             // ① 从 Indexer 删除
            handler.OnDelete(obj)               // ② 通知 sharedProcessor
        }
    }
}
```

对于首次 List 的 Pod `default/nginx`，因为 Indexer 里还没有，所以走 Add 路径：

```
① clientState.Add(obj)
   → Indexer.cacheStorage.Add("default/nginx", Pod)
   → threadSafeMap.Update()
     → items["default/nginx"] = Pod
     → index.updateIndices()  更新 namespace 索引

② handler.OnAdd(obj, true)
   → sharedIndexInformer.OnAdd()
```

## 第五阶段：事件分发 — sharedProcessor → processorListener

### 9. sharedIndexInformer.OnAdd → processor.distribute

```go
// shared_informer.go:644-649
func (s *sharedIndexInformer) OnAdd(obj interface{}, isInInitialList bool) {
    s.cacheMutationDetector.AddObject(obj)
    s.processor.distribute(
        addNotification{newObj: obj, isInInitialList: true},
        false,  // sync=false，非 Sync 事件，发给所有 listener
    )
}

// shared_informer.go:774-790
func (p *sharedProcessor) distribute(obj interface{}, sync bool) {
    for listener, isSyncing := range p.listeners {
        if !sync {
            listener.add(obj)   // ← 发给你的 listener
        }
    }
}
```

### 10. listener.add() → addCh

```go
// shared_informer.go:924-929
func (p *processorListener) add(notification interface{}) {
    p.addCh <- notification   // 阻塞发送到 addCh（无缓冲 channel）
}
```

**注意**：这一步是在持有 DeltaFIFO 锁的情况下阻塞发送的！如果你的 listener 处理太慢，addCh 没人接收，整个 DeltaFIFO 就卡住了。但因为有 `pop()` goroutine 在持续消费，所以实际上消息会很快被搬到 `pendingNotifications` 缓冲区。

### 11. pop() goroutine — 从 addCh 搬到 nextCh

```go
// shared_informer.go:931-959
func (p *processorListener) pop() {
    defer close(p.nextCh)

    var nextCh chan<- interface{}
    var notification interface{}
    for {
        select {
        case nextCh <- notification:              // 往 run() 发
            notification, ok = p.pendingNotifications.ReadOne()
            if !ok { nextCh = nil }              // 缓冲区空了
        case n, ok := <-p.addCh:                  // 从 distribute() 收
            if !ok { return }
            if notification == nil {
                // 快速路径：缓冲区空，直接转发，不经 RingGrowing
                notification = n
                nextCh = p.nextCh
            } else {
                // 慢路径：run() 还没消费完，先存到缓冲区
                p.pendingNotifications.WriteOne(n)
            }
        }
    }
}
```

对于第一个事件，走快速路径：

```
addCh 收到 addNotification
  → notification = addNotification
  → nextCh = p.nextCh
  → 下一轮循环: nextCh <- notification  （发给 run）
```

### 12. run() goroutine — 调用你的 Handler！

```go
// shared_informer.go:961-986
func (p *processorListener) run() {
    stopCh := make(chan struct{})
    wait.Until(func() {
        for next := range p.nextCh {
            switch notification := next.(type) {
            case addNotification:
                p.handler.OnAdd(notification.newObj, notification.isInInitialList)
                //      ↑                              ↑
                //  你的 ResourceEventHandlerFuncs      Pod 对象
                //  AddFunc: func(obj interface{}) { fmt.Println("Pod added!") }
            case updateNotification:
                p.handler.OnUpdate(notification.oldObj, notification.newObj)
            case deleteNotification:
                p.handler.OnDelete(notification.oldObj)
            }
        }
        close(stopCh)
    }, 1*time.Second, stopCh)
}
```

🎉 你的 `AddFunc` 终于被调用了！打印出 `"Pod added!"`

## 第六阶段：后续增量 Watch 事件

首次 List 完毕后，Reflector 进入 Watch 阶段。假设此时有人在集群里创建了一个新 Pod `default/web`：

### 13. watchHandler 接收 Watch 事件

```go
// reflector.go:717-757
case event, ok := <-w.ResultChan():    // API Server 推送 watch 事件
    switch event.Type {
    case watch.Added:
        store.Add(event.Object)        // store 就是 DeltaFIFO
```

```
          → run() → handler.OnAdd()         // 🎉 你的 Handler 再次被调用
```

## WorkQueue vs pendingNotifications

### 简短回答

不是。它们虽然都在"缓冲"，但定位完全不同。WorkQueue 远比 `pendingNotifications` 复杂得多，它解决的核心问题是 **去重 + 并发控制 + 重试 + 限流**，不是简单的缓冲。

### 先看各自长什么样

#### pendingNotifications — 无界环形缓冲区

```go
// shared_informer.go:872
// pendingNotifications is an unbounded ring buffer that holds all
// notifications not yet distributed.
pendingNotifications buffer.RingGrowing
```

- **位置**：`processorListener` 内部，夹在 `pop()` 和 `nextCh` 之间
- **内容**：notification 对象（`addNotification` / `updateNotification` / `deleteNotification`）
- **消费者**：只有 1 个 goroutine（`run()`）
- **容量**：无界，可以无限增长
- **去重**：无，每个事件都存

#### WorkQueue — 带状态机的工作队列

```go
// queue.go:115-138
type Type struct {
    queue      []t    // 有序队列，等待处理
    dirty      set    // 所有需要处理的 item（去重用）
    processing set    // 正在处理中的 item
    cond       *sync.Cond
    ...
}
```

- **位置**：Controller 层，夹在 EventHandler 和 worker goroutine 之间
- **内容**：Key 字符串（如 `"default/nginx"`），不是完整对象
- **消费者**：N 个 worker goroutine 并发消费
- **去重**：有，同一个 Key 短时间多次 Add 只入队一次
- **容量**：有界
- **重试**：有，失败可以 `AddRateLimited` 延迟重试
- **限流**：有，`RateLimiter` 控制重试间隔

### 核心区别：4 个维度

```
                  pendingNotifications              WorkQueue
                  ─────────────────────             ──────────────────────────
定位              内部解耦，保护 Informer            业务解耦，保护 Controller
                  分发不阻塞                        调谐逻辑不阻塞事件分发

去重              无                                ★ 有（dirty set）
                  同一个对象变了3次 → 3个通知         同一个 Key Add 3次 → 只处理 1 次

并发              单消费者（1个 run() goroutine）     多消费者（N 个 worker）
                                                    processing set 保证同一个 Key
                                                    不会被两个 worker 同时处理

重试              无                                ★ 有（AddRateLimited + RateLimiter）
                  处理失败就丢了                      失败 → 退避重试 → 指数退避
```

### 用一个具体场景说明

假设 `default/nginx` Pod 在短时间内发生了 100 次状态变化：

#### 在 processorListener 这一层（pendingNotifications）

```
Watch 事件 #1: Pod{v1}   → notification → addCh → pop() → pendingNotifications → run() → OnAdd
Watch 事件 #2: Pod{v2}   → notification → addCh → pop() → pendingNotifications → run() → OnUpdate
Watch 事件 #3: Pod{v3}   → notification → addCh → pop() → pendingNotifications → run() → OnUpdate
...
Watch 事件 #100: Pod{v100} → notification → addCh → pop() → pendingNotifications → run() → OnUpdate
```

**结果**：100 个事件，100 次通知，你的 Handler 被调用了 100 次

#### 在 Controller 这一层（WorkQueue）

```go
// 你的 Handler 里：
func (c *Controller) enqueueFoo(obj interface{}) {
    key, _ := cache.MetaNamespaceKeyFunc(obj)
    c.workqueue.Add(key)  // ← 注意：这里调的是 WorkQueue.Add
}
```

```
事件 #1:  Add("default/nginx")  → dirty.insert, queue=[key]     → worker 取走 → processing
事件 #2:  Add("default/nginx")  → dirty 已有，直接跳过！          worker 还在处理 #1
事件 #3:  Add("default/nginx")  → dirty 已有，直接跳过！
...
事件 #100: Add("default/nginx") → dirty 已有，直接跳过！
...
worker 完成 #1: Done("default/nginx")
              → dirty 有吗？有！（中间 Add 过）
              → 重新入队！queue=[key]
              → worker 取走，syncHandler("default/nginx")
              → 读最新状态 Pod{v100}，直接调谐
```

**结果**：100 次变化，Worker 只调谐了 2 次（第1次 + Done后发现dirty再调谐1次）

这就是 WorkQueue 的威力：无论事件多密集，同一个 Key 的调谐只发生必要的次数。

### 去重是怎么实现的

WorkQueue 的 `Add`（`queue.go:163-182`）：

```go
func (q *Type) Add(item interface{}) {
    q.cond.L.Lock()
    defer q.cond.L.Unlock()

    if q.dirty.has(item) {
        return          // ★ 已经在 dirty 里了，跳过！
    }

    q.dirty.insert(item)
    if q.processing.has(item) {
        return          // ★ 正在被 worker 处理，不重复入队（Done 时会重新检查 dirty）
    }

    q.queue = append(q.queue, item)  // 入队
    q.cond.Signal()                   // 唤醒一个等待的 worker
}
```

`Done`（`queue.go:223-236`）：

```go
func (q *Type) Done(item interface{}) {
    q.processing.delete(item)

    if q.dirty.has(item) {           // 处理期间又有人 Add 过？
        q.queue = append(q.queue, item)  // ★ 重新入队！
        q.cond.Signal()
    }
}
```

三个 set 的状态机：

```
    Add(item)
        │
        ▼
    ┌───────┐    Get()     ┌────────────┐   Done()
    │ dirty │──────────────│ processing │──────────┐
    │ + queue│             │            │          │
    └───────┘             └────────────┘          │
        │                                          │
        │  处理期间又 Add()                          │
        │  dirty 里有 → Done 时重新入队              │
        └──────────────────────────────────────────┘
```

`pendingNotifications` 完全没有这套机制，它就是个 FIFO 缓冲区。

## 两个 Handler 的区分

源码里到处都是 "handler" 这个词，但它们其实是不同层次的东西。

### 回答核心问题

`AddEventHandler` 注册的是外部 handler，也就是你写的 Controller 逻辑。而 `processorListener.handler` 存的就是这个外部 handler。

但与此同时，`sharedIndexInformer` 自己也实现了 `ResourceEventHandler` 接口，作为内部 handler 接收 `processDeltas` 的调用并转发给所有 listener。

这两个 "handler" 是不同层次的东西，但共享同一个接口类型。

### 两个 Handler 的对比

```
┌─────────────────────────────────────────────────────────────────┐
│ 接口: ResourceEventHandler                                       │
│   OnAdd(obj, isInInitialList)                                   │
│   OnUpdate(old, new)                                            │
│   OnDelete(obj)                                                 │
└─────────────────────────────────────────────────────────────────┘
         ▲                                    ▲
         │ 实现了                               │ 实现了
         │                                    │
┌────────┴──────────┐              ┌──────────┴──────────────────┐
│ sharedIndexInformer│              │ ResourceEventHandlerFuncs   │
│  (内部 handler)    │              │  (外部 handler / 你写的)     │
│                    │              │                             │
│ OnAdd() → 转发给   │              │ AddFunc: enqueueFoo         │
│   sharedProcessor  │              │ UpdateFunc: enqueueFoo      │
│                    │              │ DeleteFunc: handleObject    │
└────────────────────┘              └─────────────────────────────┘
     负责"接收+转发"                      负责"业务逻辑（入队等）"
```

### 完整调用链

```
processLoop
  │ Pop() → Deltas
  ▼
HandleDeltas()                              ← sharedIndexInformer 的方法
  │
  ▼
processDeltas(s, s.indexer, deltas)
  │
  │ handler = s (sharedIndexInformer)
  │
  ▼
handler.OnAdd(obj)                          ← ★ 调内部 handler（sharedIndexInformer）
  │
  │  内部 handler 做的事：转发给 sharedProcessor
  ▼
sharedProcessor.distribute(addNotification)
  │ 遍历所有 listener
  ▼
listener.add(obj)                           ← addCh
  │
  ▼
pop() 搬运 → nextCh
  │
  ▼
run() → p.handler.OnAdd(obj)                ← ★ 调外部 handler（你写的）
  │
  │  外部 handler 做的事：workqueue.Add(key)
  ▼
WorkQueue → Worker → syncHandler(key)       ← 真正的调谐逻辑
```
