## 0. 本课目标

本课目标：

```text
看到 Pod 重启、OOMKilled、节点异常、kubelet/containerd 异常时，
能把 Kubernetes 层事件和 Linux 节点日志对齐起来。
```

核心方法：

```text
不要只看一层日志。
按时间线把 K8s Event、kubelet 日志、containerd 日志、内核日志串起来。
```

---

## 1. 三层日志入口

### Kubernetes 对象层

常用命令：

```bash
kubectl describe pod <pod>
kubectl get events --sort-by=.lastTimestamp
kubectl get events -A --sort-by=.lastTimestamp | tail -30
```

主要回答：

```text
Kubernetes 认为发生了什么。
Pod 是否调度成功。
容器状态是什么。
Last State / Reason / Exit Code 是什么。
Events 中 kubelet、scheduler、controller 上报了什么。
```

### 节点 systemd 服务层

kubelet：

```bash
sudo journalctl -u kubelet --since "1 hour ago" --no-pager | tail -100
sudo journalctl -u kubelet -f
```

主要回答：

```text
kubelet 在节点上做了什么。
为什么拉镜像失败。
为什么创建容器失败。
为什么上报 Pod 状态。
是否有 eviction、volume、PLEG、runtime 相关错误。
```

containerd：

```bash
sudo journalctl -u containerd --since "1 hour ago" --no-pager | tail -100
sudo journalctl -u containerd -f
```

主要回答：

```text
容器运行时创建、启动、停止、清理容器时发生了什么。
containerd-shim、task、snapshot、CNI 调用是否异常。
```

### 内核层

常用命令：

```bash
sudo dmesg -T | tail -100
sudo journalctl -k --since "1 hour ago" --no-pager | tail -100
```

主要回答：

```text
内核发生了什么。
是否有 OOM。
是否有磁盘 IO 错误。
是否有文件系统错误。
是否有网卡、驱动、cgroup kill 等事件。
```

---

## 2. 常用过滤命令

kubelet：

```bash
sudo journalctl -u kubelet --since "1 hour ago" --no-pager \
  | grep -i -E 'error|fail|oom|kill|evict|image|pull|container'
```

containerd：

```bash
sudo journalctl -u containerd --since "1 hour ago" --no-pager \
  | grep -i -E 'error|fail|shim|task|oom|kill'
```

内核：

```bash
sudo journalctl -k --since "1 hour ago" --no-pager \
  | grep -i -E 'oom|killed process|memory cgroup|disk|ext4|xfs|io error'
```

## 3. 事件对齐方法

通用排查顺序：

```text
1. kubectl get pod 看现象。
2. kubectl describe pod 看 State / Last State / Reason / Events。
3. 找 Pod 所在 Node。
4. 到 Node 上查 kubelet 日志。
5. 查 containerd 日志。
6. 如果涉及 OOM、IO、文件系统、网卡、cgroup，再查 dmesg 或 journalctl -k。
```

示例：OOMKilled

```text
kubectl describe pod:
Reason: OOMKilled
Exit Code: 137

dmesg / journalctl -k:
Memory cgroup out of memory
Killed process <pid> (<name>)
constraint=CONSTRAINT_MEMCG
```

示例：镜像拉取失败

```text
kubectl describe pod:
ErrImagePull / ImagePullBackOff

journalctl -u kubelet:
PullImage from image service failed
Failed to pull image
rpc error
not found / unauthorized / timeout / x509
```

示例：容器启动失败

```text
kubectl describe pod:
CreateContainerError / CrashLoopBackOff

journalctl -u containerd:
failed to create shim task
OCI runtime create failed
```

## 4. 常规观察实验

控制端观察：

```bash
kubectl get events -A --sort-by=.lastTimestamp | tail -30
kubectl get pod -A -o wide
```

实验输出节选：

```text
default      Normal   RegisteredNode  node/k8s-node2
kube-system  Warning  Unhealthy       pod/calico-node-6kn8r
Readiness probe failed: calico/node is not ready:
BIRD is not ready:
Error querying BIRD:
unable to connect to BIRDv4 socket:
dial unix /var/run/calico/bird.ctl: connect: connection refused
```

node2 kubelet 日志节选：

```text
Starting Kubelet Volume Manager
Error getting the current node from lister err="node \"k8s-node2\" not found"
```

解读：

```text
这通常出现在节点或 kubelet 刚启动、还没有完成和 apiserver 状态同步时。
随后 RegisteredNode 出现，说明节点注册成功。
```

Calico 事件：

```text
BIRD is not ready
unable to connect to BIRDv4 socket
```

解读：

```text
如果只在启动早期出现，常见原因是 calico-node 启动过程中 BIRD 进程或 socket 尚未就绪。
如果后续 calico-node Ready，且 BGP 179 连接 ESTAB，通常不是持续故障。
```

之前已观察到 node2 上 BGP 连接：

```text
192.168.59.136 -> 192.168.59.134:179 ESTAB
192.168.59.136 -> 192.168.59.135:179 ESTAB
```

说明后续 BGP session 已正常建立。

## 5. containerd/CNI 清理日志

containerd 日志节选：

```text
Cleaning up netns
CleanUpNamespace called with no netns name, ignoring
Releasing IP address(es)
Calico CNI releasing IP address
```

解读：

```text
containerd/CNI 正在清理某个容器网络命名空间并释放 Pod IP。
这通常发生在 Pod 删除、容器重建、节点启动后清理残留状态时。
```

## 6. 内核日志观察

命令：

```bash
sudo journalctl -k --since "2 hours ago" --no-pager \
  | grep -i -E 'oom|killed process|memory cgroup|disk|ext4|xfs|io error|calico|tunl|eth|ens33' \
  | tail -80
```

输出节选：

```text
RAMDISK: [mem ...]
VFS: Disk quotas dquot_6.6.0
e1000 ... eth0: Intel(R) PRO/1000 Network Connection
sd ... [sda] Attached SCSI disk
e1000 ... ens33: renamed from eth0
```

解读：

```text
这是系统启动时硬件、网卡、磁盘初始化信息。
当前没有看到 OOM、磁盘 IO 错误、文件系统错误。
```

创建一个不存在镜像 tag 的 Pod：

```bash
kubectl run bad-image --image=nginx:not-exist-20260614 --restart=Never
```

```bash
kubectl get pod bad-image -w
kubectl describe pod bad-image
```

## 10. kubelet、CRI 与 containerd 的关系

错误中有：

```text
rpc error: code = NotFound
```

说明 kubelet 通过 CRI 调用 image service 拉镜像，底层 containerd 返回了 NotFound 错误。

链路：

```text
kubelet
  -> CRI ImageService.PullImage
    -> containerd
      -> registry
        -> NotFound
  <- kubelet 记录日志并上报 Event
  <- Pod 状态变成 ErrImagePull / ImagePullBackOff
```

---

## 11. ErrImagePull 与 ImagePullBackOff

`ErrImagePull`：

```text
一次镜像拉取失败的直接错误。
```

`ImagePullBackOff`：

```text
kubelet 在拉取失败后进入退避重试状态。
不会无脑连续拉取，而是间隔越来越长再尝试。
```

所以看到：

```text
Pulling image x2
Back-off pulling image
```

是正常重试逻辑。

---

## 12. 镜像拉取失败分类

看到 `ErrImagePull` / `ImagePullBackOff` 时，优先看 Message。

### tag 或镜像不存在

典型信息：

```text
not found
failed to resolve reference
```

本次实验就是：

```text
docker.io/library/nginx:not-exist-20260614: not found
```

判断：

```text
镜像名或 tag 不存在。
```

### 认证失败

典型信息：

```text
authentication required
unauthorized
denied
```

优先检查：

```text
imagePullSecrets
ServiceAccount 是否挂载了正确 secret
镜像仓库账号权限
secret 所在 namespace 是否正确
```

### 网络超时

典型信息：

```text
i/o timeout
TLS handshake timeout
context deadline exceeded
```

优先检查：

```text
节点到镜像仓库的网络连通性
DNS
代理
防火墙
镜像仓库可用性
```

### 证书问题

典型信息：

```text
x509 certificate signed by unknown authority
```

优先检查：

```text
私有仓库证书
containerd registry 配置
节点信任链
```

### DNS 问题

典型信息：

```text
no such host
```

优先检查：

```text
节点 DNS
/etc/resolv.conf
systemd-resolved
上游 DNS
```

## 14. 小测

问题 1：`kubectl describe pod`、`journalctl -u kubelet`、`journalctl -k` 分别主要看哪一层的问题？

回答：

```text
kubectl describe pod 看 k8s 层信息。
journalctl -u kubelet 看节点层信息。
journalctl -k 看内核层信息。
```

点评：正确。

补充：

```text
journalctl -u kubelet 更准确是看节点上的 kubelet 服务层日志。
节点层还可能包括 containerd、systemd、网络服务等。
```

问题 2：`ErrImagePull` 和 `ImagePullBackOff` 有什么区别？

回答：

```text
ErrImagePull 是某次拉取失败。
ImagePullBackOff 是拉取失败后触发退避重试拉取。
```

点评：正确。

问题 3：这次 `bad-image` 为什么不是调度失败？

回答：

```text
可以看到已经调度到 node1 并且分配了 pod ip。
```

点评：正确。

补充：

```text
Events 中还有 Scheduled Successfully assigned，也说明调度成功。
```

问题 4：`docker.io/library/nginx:not-exist-20260614: not found` 说明更像哪类镜像拉取问题？

回答：

```text
标签不存在问题。
```

点评：正确。

问题 5：如果日志里是 `authentication required`，你会优先检查什么？

回答：

```text
认证错误，检查秘钥相关认证。
```

点评：正确。

补充：

```text
重点检查 imagePullSecrets、ServiceAccount、secret namespace、仓库权限。
```

问题 6：如果 Pod 是 `OOMKilled`，除了 `kubectl describe pod`，节点上应该查什么日志？

回答：

```text
节点上应该看内核级的日志，是否触发 cgroup oom。
```

点评：正确。

命令：

```bash
sudo dmesg -T | grep -i -E 'oom|killed process|memory cgroup' | tail -50
sudo journalctl -k --since "1 hour ago" --no-pager | grep -i -E 'oom|killed process|memory cgroup'
```

问题 7：Calico `BIRD is not ready` 如果只在启动早期出现，后面 BGP 179 连接正常，通常说明什么？

回答：

```text
通常是节点刚启动的日志情况。
```

点评：正确。

补充：

```text
更准确地说，是 calico-node/BIRD 启动早期尚未就绪的瞬态状态。
如果后续 Ready 且 BGP ESTAB，一般不是持续故障。
```

## 15. 必须记牢的短句

```text
kubectl describe 看 K8s 认为发生了什么。
journalctl -u kubelet 看 kubelet 在节点上做了什么。
journalctl -u containerd 看容器运行时做了什么。
journalctl -k / dmesg 看内核发生了什么。
排障时按时间线对齐，不要只看一层日志。
ErrImagePull 是一次拉取失败。
ImagePullBackOff 是拉取失败后的退避重试状态。
not found 通常是镜像名或 tag 不存在。
authentication required 优先查 imagePullSecrets 和仓库权限。
OOMKilled 要去内核日志查 memory cgroup out of memory。
Calico BIRD 启动早期不 ready，后续 BGP ESTAB，通常是瞬态启动状态。
```
