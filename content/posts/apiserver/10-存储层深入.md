## 前言

第四篇介绍了 `genericregistry.Store` 和 Strategy 的基本概念。这篇从底层接口开始，一步步深入到 Store 内部实现，再到具体资源（Service）的组装，最后看 Store 是怎么变成 HTTP 路由的。

先建立心智模型：K8s API Server 本质上是一个**资源的 CRUD 服务**。它不是为某一种资源写代码，而是让开发者用最少的代码注册一种新资源。"注册一种新资源"被拆成三件事：

1. 这个资源长什么样？→ Type（Go struct，scheme 注册）
2. 这个资源的 CRUD 怎么做？→ Storage（通用 Store + etcd）
3. 这个资源有什么业务规则？→ Strategy（校验、默认值、字段重置等）

## 第一层：接口层 — "你能对一个资源做什么"

`rest.go` 定义了一组接口，每个接口代表一种 REST 操作能力。

### Storage — 最基础的接口

```go
type Storage interface {
    New() runtime.Object   // 给我一个空对象，用于反序列化请求体
    Destroy()              // 关机时清理资源
}
```

为什么 `New()` 必须返回指针？因为 K8s 的 Codec 需要往里面写数据（`DecodeInto`），Go 中只有指针才能被修改。和 `json.Unmarshal(data, &obj)` 是一个道理。

### Scoper — 命名空间作用域

```go
type Scoper interface {
    NamespaceScoped() bool  // true = Pod（namespace 下），false = Node（集群级别）
}
```

这决定了 URL 路径：`/api/v1/namespaces/{ns}/pods` vs `/api/v1/nodes`。

### CRUD 操作接口 — 一个操作一个接口

K8s 把每个 REST 操作拆成独立的接口：

| 接口 | 对应 HTTP | 方法签名 |
|------|-----------|----------|
| `Getter` | `GET /pods/{name}` | `Get(ctx, name, opts) → Object` |
| `Lister` | `GET /pods` | `List(ctx, opts) → Object` |
| `Creater` | `POST /pods` | `Create(ctx, obj, validation, opts) → Object` |
| `Updater` | `PUT /pods/{name}` | `Update(ctx, name, objInfo, ...) → Object` |
| `GracefulDeleter` | `DELETE /pods/{name}` | `Delete(ctx, name, validation, opts) → Object, bool` |
| `CollectionDeleter` | `DELETE /pods` | `DeleteCollection(ctx, ...) → Object` |
| `Watcher` | `GET /pods?watch=true` | `Watch(ctx, opts) → watch.Interface` |

**为什么不像普通后端那样用一个 CRUD 接口？** 因为不是每种资源都支持所有操作。比如 events 只允许 List/Get，不允许 Update。拆开后，API Server 在安装路由时可以做接口断言——你实现了哪个接口，就注册哪个 endpoint，没实现的自动跳过。这是一种**按能力组合**的设计模式。

### StandardStorage — 全套餐

```go
type StandardStorage interface {
    Getter
    Lister
    CreaterUpdater    // 注意：不是 Creater + Updater，而是合并版
    GracefulDeleter
    CollectionDeleter
    Watcher
    Destroy()
}
```

`genericregistry.Store` 就是它的实现。

为什么 `CreaterUpdater` 要单独存在？因为 Go 不允许同时嵌入两个有同名方法的接口（`Creater` 和 `Updater` 都有 `New()` 方法），所以必须合并成一个。

### Patcher — PATCH 操作的组合

```go
type Patcher interface {
    Getter    // 先 GET 当前对象
    Updater   // 再 UPDATE 修改后的对象
}
```

PATCH 的实现逻辑就是：先读出来，应用 patch，再写回去。

### 接口层全景

```
Storage（最基础）
  ├── Scoper（namespace 作用域）
  ├── Getter / Lister / Creater / Updater / Deleter / Watcher（每个操作一个接口）
  ├── StandardStorage（全套餐）
  └── Patcher（GET + UPDATE 的组合）
```

## 第二层：Strategy 层 — "这个资源有什么业务规则"

接口层解决了"能做什么操作"，但每个资源的业务规则不同。比如 Pod 创建时要校验容器镜像不能为空，Deployment 更新时 `spec.selector` 不能改，Status 更新时不能改 Spec。

K8s 把它们抽象成 Strategy 模式——你只需要填几个"回调函数"，通用的 Store 会在合适的时机调用它们。

三个 Strategy 接口分别对应三种操作：

```
RESTCreateStrategy  →  创建时的规则
RESTUpdateStrategy  →  更新时的规则
RESTDeleteStrategy  →  删除时的规则
```

### RESTCreateStrategy

```go
type RESTCreateStrategy interface {
    runtime.ObjectTyper          // 能识别对象的 GVK
    names.NameGenerator          // 能生成名字（当用户用 generateName 时）

    NamespaceScoped() bool
    PrepareForCreate(ctx, obj)   // 创建前的"清洗"
    Validate(ctx, obj)           // 校验
    WarningsOnCreate(ctx, obj)   // 返回警告
    Canonicalize(obj)            // 规范化
}
```

**为什么要有 `PrepareForCreate`？** 在验证之前清洗数据。最常见的用法是清除 Status——用户不应该在创建时设置 Status，Status 是由控制器计算出来的。如果不清除，恶意用户可以在创建时把 Status 设成一个假的"已就绪"状态。

用 Deployment 举个例子：

```go
func (deploymentStrategy) PrepareForCreate(ctx context.Context, obj runtime.Object) {
    deployment := obj.(*apps.Deployment)
    deployment.Status = apps.DeploymentStatus{}   // 1. 清掉 Status
    deployment.Generation = 1                       // 2. 设置 Generation 初始值
    pod.DropDisabledTemplateFields(...)             // 3. 去掉未启用的 feature 字段
}
```

调用时机（`BeforeCreate` 函数）：

```
1. 检查 metadata 系统字段已初始化
2. 检查 name 已生成（如果用了 generateName）
3. 检查 namespace 匹配
4. PrepareForCreate()     ← 先清洗
5. Validate()             ← 再校验
6. ValidateObjectMeta()   ← 校验 metadata 通用字段
7. WarningsOnCreate()     ← 返回警告
8. Canonicalize()         ← 最后规范化
```

**顺序很重要：先清洗再校验，否则你校验的可能是"脏数据"。**

### RESTUpdateStrategy

```go
type RESTUpdateStrategy interface {
    runtime.ObjectTyper
    NamespaceScoped() bool
    AllowCreateOnUpdate() bool        // PUT 一个不存在的对象时，是否自动创建？
    PrepareForUpdate(ctx, obj, old)   // 清洗，注意有 old 参数
    ValidateUpdate(ctx, obj, old)     // 校验，也有 old
    WarningsOnUpdate(ctx, obj, old)
    Canonicalize(obj)
    AllowUnconditionalUpdate() bool   // 不指定 resourceVersion 也能更新？
}
```

**为什么 `PrepareForUpdate` 有 `old` 参数而 `PrepareForCreate` 没有？** 因为更新需要保护不变量。看 Deployment 的实现：

```go
func (deploymentStrategy) PrepareForUpdate(ctx context.Context, obj, old runtime.Object) {
    newDeployment := obj.(*apps.Deployment)
    oldDeployment := old.(*apps.Deployment)

    // 1. 保护 Status：用户通过 spec 更新时不能改 status
    newDeployment.Status = oldDeployment.Status

    // 2. 保护 Generation：只有 spec 或 annotations 变了才递增
    if !apiequality.Semantic.DeepEqual(newDeployment.Spec, oldDeployment.Spec) ||
       !apiequality.Semantic.DeepEqual(newDeployment.Annotations, oldDeployment.Annotations) {
        newDeployment.Generation = oldDeployment.Generation + 1
    }
}
```

为什么 Generation 的递增是条件性的？Deployment Controller 监听 Spec 变化来触发滚动更新。如果每次 status 更新也递增 Generation，Controller 就会误以为 Spec 变了，导致无限循环滚动更新。

### RESTDeleteStrategy

```go
type RESTDeleteStrategy interface {
    runtime.ObjectTyper
}
```

极简！只有一个类型标识。真正的删除逻辑在 `BeforeDelete` 函数里处理，它做了一件很重要的事——优雅删除（Graceful Deletion）。

K8s 的删除分两步：

```
第一步：设置 DeletionTimestamp（标记为"要删了"）
       → 控制器看到这个标记，开始清理（比如从 Service 摘除 endpoint）
       → Pod 里的容器收到 SIGTERM，开始优雅关闭

第二步：GracePeriod 到期后，真正从 etcd 删掉
```

还有两个可选接口扩展删除行为：
- `RESTGracefulDeleteStrategy` — 控制是否支持优雅删除
- `GarbageCollectionDeleteStrategy` — 控制级联删除策略

### Status Strategy — 子资源的特殊规则

K8s 有一个 `/status` 子资源端点：

```
PUT /apis/apps/v1/namespaces/default/deployments/my-app         → 更新 spec
PUT /apis/apps/v1/namespaces/default/deployments/my-app/status  → 更新 status
```

这两个端点用不同的 Strategy！Status Strategy 嵌入主 Strategy，只覆盖 `PrepareForUpdate`：

```go
// 翻转保护方向：status 更新时保护 spec 和 labels
func (deploymentStatusStrategy) PrepareForUpdate(ctx context.Context, obj, old runtime.Object) {
    newDeployment := obj.(*apps.Deployment)
    oldDeployment := old.(*apps.Deployment)
    newDeployment.Spec = oldDeployment.Spec      // spec 不许改
    newDeployment.Labels = oldDeployment.Labels  // labels 不许改
}
```

对比一下：

| 操作 | 主 Strategy 的 PrepareForUpdate | Status Strategy 的 PrepareForUpdate |
|------|---------------------------------|-------------------------------------|
| 保护对象 | Status（用 old 覆盖 new） | Spec + Labels（用 old 覆盖 new） |
| 允许修改 | Spec, Labels, Annotations | Status, Annotations |

**为什么不用校验来拒绝非法修改，而用 Prepare 直接覆盖？** 用 Validate 的话，用户发了一个同时改了 spec 和 status 的请求会直接报错。但用 Prepare 覆盖后，spec 部分被静默忽略，status 部分正常更新。这是一种"部分更新，忽略不允许的字段"的策略，对用户更友好。

### Strategy 在整个流程中的位置

```
用户请求 → HTTP Handler → BeforeCreate/BeforeUpdate/BeforeDelete
                               │
                               ├─ PrepareForXxx()   ← 清洗数据
                               ├─ Validate()         ← 校验数据
                               ├─ WarningsOnXxx()   ← 返回警告
                               ├─ Canonicalize()     ← 规范化
                               │
                               └─ 写入 etcd（由 Store 完成）
```

Strategy 不直接操作 etcd，它只负责"数据进入 etcd 前的规则"。通用流程（Store）不变，具体规则（Strategy）可替换——策略模式的经典应用。

## 第三层：Store 结构体 — "通用 CRUD 的发动机"

Store 是 K8s Storage 层的核心。用一套通用代码实现所有资源的 CRUD + Watch 操作，具体资源的差异通过注入字段来定制。这是**模板方法模式**——流程固定，细节可替换。

### 字段分类

Store 有 28 个字段，分成 5 大类：

**对象工厂 — "怎么创建空对象"**

```go
NewFunc     func() runtime.Object  // 返回一个空的单对象，如 &Pod{}
NewListFunc func() runtime.Object  // 返回一个空的列表对象，如 &PodList{}
```

为什么需要工厂函数而不是直接传类型？因为 Go 早期没有泛型，而且 K8s 的对象都是 `runtime.Object` 接口，需要用工厂函数来创建具体类型。

**etcd Key 生成 — "数据存在哪里"**

```go
KeyRootFunc    func(ctx) string                // 集合的根 key，如 /registry/pods/default
KeyFunc        func(ctx, name) (string, error) // 单个对象的 key，如 /registry/pods/default/my-pod
ObjectNameFunc func(obj) (string, error)       // 从对象中提取名字
```

默认实现根据是否命名空间自动设置：

```go
// 命名空间资源：/registry/pods/{namespace}/{name}
func NamespaceKeyFunc(ctx, prefix, name) (string, error) {
    ns, _ := NamespaceFrom(ctx)
    return prefix + "/" + ns + "/" + name, nil
}

// 集群级资源：/registry/nodes/{name}
func NoNamespaceKeyFunc(ctx, prefix, name) (string, error) {
    return prefix + "/" + name, nil
}
```

**Strategy — "业务规则"**

```go
CreateStrategy rest.RESTCreateStrategy
UpdateStrategy rest.RESTUpdateStrategy
DeleteStrategy rest.RESTDeleteStrategy
```

**生命周期钩子 — "操作前后想做什么"**

```go
BeginCreate  BeginCreateFunc   // 创建前，返回 commit/rollback 函数
AfterCreate  AfterCreateFunc   // 创建后
BeginUpdate  BeginUpdateFunc   // 更新前，返回 commit/rollback 函数
AfterUpdate  AfterUpdateFunc   // 更新后
AfterDelete  AfterDeleteFunc   // 删除后
Decorator    func(Object)      // 读取后装饰（所有 GET/List/Watch 返回的对象都会经过它）
```

`BeginCreate` 为什么要返回 `FinishFunc`？这是一个"事务式"钩子。`BeginCreate` 在操作前调用，返回一个 `FinishFunc`。操作成功后调用 `FinishFunc(ctx, true)`，失败后调用 `FinishFunc(ctx, false)`。这样你可以在 `BeginCreate` 中锁定资源，在 `FinishFunc` 中根据成功/失败决定提交还是回滚。

**底层存储 — "数据怎么读写"**

```go
Storage          DryRunnableStorage   // 实际的 etcd 操作封装
StorageVersioner runtime.GroupVersioner // 对象序列化到 etcd 时用哪个版本
DestroyFunc      func()               // 关机时清理 etcd 客户端
```

`DryRunnableStorage` 在外面包了一层，支持 dry-run（模拟操作但不真正写入）。

### 核心方法的执行流程

**Create 方法 — 完整流程**

```
1. 初始化 metadata
   ├─ FillObjectMetaSystemFields()    // 设置 CreationTimestamp、UID
   └─ GenerateName()                  // 如果用了 generateName，生成名字

2. BeginCreate 钩子（如果有的话）
   └─ 返回 FinishFunc，defer 注册失败时自动调用

3. BeforeCreate(e.CreateStrategy, ctx, obj)
   ├─ PrepareForCreate()              // 清洗（清 Status 等）
   ├─ Validate()                      // 校验
   └─ Canonicalize()                  // 规范化

4. createValidation()                 // Admission 链的校验

5. 计算 etcd key 和 TTL

6. e.Storage.Create(key, obj, out, ttl)  ← 写入 etcd

7. 成功后：
   ├─ FinishFunc(ctx, true)           // 提交
   ├─ AfterCreate()                   // 后置钩子
   └─ Decorator()                     // 装饰返回值
```

写入 etcd 失败时，会尝试 GET 已有对象。如果发现对象正在被删除（`DeletionTimestamp != nil`），错误信息会说"object is being deleted"而不是简单的"AlreadyExists"。

**Update 方法 — 最复杂的方法**

核心是一个 `GuaranteedUpdate` 调用，它是一个 CAS（Compare-And-Swap）循环：

```go
err = e.Storage.GuaranteedUpdate(ctx, key, out, true, preconditions,
    func(existing Object, res ResponseMeta) (Object, *uint64, error) {
        // 这个函数可能被调用多次（乐观锁冲突时重试）

        // 1. 检查对象是否存在
        // 2. 获取用户提交的新对象
        obj, _ := objInfo.UpdatedObject(ctx, existing)

        // 3. 乐观锁检查
        if newResourceVersion != existingResourceVersion {
            return Conflict("object has been modified")
        }

        // 4. 调用 Strategy
        BeforeUpdate(e.UpdateStrategy, ctx, obj, existing)
        updateValidation(ctx, obj, existing)

        return obj, ttl, nil
    })
```

K8s 用的是**乐观并发控制**。每个对象都有 `ResourceVersion`（本质是 etcd 的 revision）。更新时必须带上当前的 `ResourceVersion`，如果在这期间有人改了这个对象，版本号就对不上，返回 Conflict 错误。就像 Git 的"先 pull 再 push"。

**Get 方法 — 最简单的方法**

```go
func (e *Store) Get(ctx, name, options) (Object, error) {
    obj := e.NewFunc()                          // 1. 创建空对象
    key, _ := e.KeyFunc(ctx, name)              // 2. 计算 etcd key
    e.Storage.Get(ctx, key, options, obj)        // 3. 从 etcd 读取
    if e.Decorator != nil { e.Decorator(obj) }  // 4. 装饰
    return obj, nil
}
```

**Delete 方法 — 两阶段删除**

```
1. 从 etcd GET 当前对象
2. BeforeDelete() 判断是否需要优雅删除
3. 如果是优雅删除或有 finalizer：
   └─ 设置 DeletionTimestamp + DeletionGracePeriodSeconds，写回 etcd
4. 如果可以立即删除：
   └─ e.Storage.Delete()  ← 真正从 etcd 删掉
```

K8s 的删除是一个状态机转换：

```
正常运行 → 标记删除（DeletionTimestamp 设置）→ finalizer 清理 → 真正删除
```

**Watch 方法**

```go
func (e *Store) Watch(ctx, options) (watch.Interface, error) {
    predicate := e.PredicateFunc(label, field)

    // 优化：如果只 watch 一个特定对象，用精确 key
    if name, ok := predicate.MatchesSingle(); ok {
        key, _ := e.KeyFunc(ctx, name)
        // 非递归 watch，性能更好
    }

    w, _ := e.Storage.Watch(ctx, key, storageOpts)

    if e.Decorator != nil {
        return newDecoratedWatcher(ctx, w, e.Decorator), nil
    }
    return w, nil
}
```

如果你只 watch 一个特定名字的对象（如 `watch=true&fieldSelector=metadata.name=my-pod`），Store 会用精确的 etcd key 做非递归 watch，而不是 watch 整个集合再过滤。

### Store 的整体架构

```
┌─────────────────────────────────────────────────────┐
│                    Store 结构体                       │
│                                                     │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐          │
│  │ NewFunc  │  │ KeyFunc  │  │ Predicate│  ← 工厂   │
│  └──────────┘  └──────────┘  └──────────┘          │
│                                                     │
│  ┌──────────────────────────────────────┐           │
│  │ CreateStrategy / UpdateStrategy /    │  ← 业务规则│
│  │ DeleteStrategy                       │           │
│  └──────────────────────────────────────┘           │
│                                                     │
│  ┌──────────────────────────────────────┐           │
│  │ Begin/After Hooks + Decorator        │  ← 扩展点 │
│  └──────────────────────────────────────┘           │
│                                                     │
│  ┌──────────────────────────────────────┐           │
│  │ Storage (DryRunnableStorage)         │  ← etcd层 │

│  Create() | Get() | List() | Update() | Delete()    │

└─────────────────────────────────────────────────────┘
```

## 第四层：Service 的 Store — 一个具体资源的完整组装

选 Service 做例子，因为它有独特的设计点（ClusterIP 分配），但 Store 本身不会太复杂。

### 整体结构

```
┌───────────────────────────────────────────────────┐
│                   REST struct                      │
│                                                   │
│  ┌─────────────────────────────────┐              │
│  │     *genericregistry.Store      │ ← 通用 CRUD  │
│  └─────────────────────────────────┘              │
│                                                   │
│  primaryIPFamily   api.IPFamily    ← 自定义字段    │
│  secondaryIPFamily api.IPFamily                    │
│  alloc             Allocators      ← IP/端口分配器 │
│  endpoints         EndpointsStorage ← 联动删除     │
│  pods              PodStorage                      │
│  proxyTransport    http.RoundTripper               │
└───────────────────────────────────────────────────┘
```

REST struct 嵌入 `*genericregistry.Store`，获得所有通用 CRUD 能力，然后用自定义字段和钩子覆盖 Service 特有的行为。

### 第一步：创建 Store

```go
store := &genericregistry.Store{
    // ── 工厂函数 ──
    NewFunc:     func() runtime.Object { return &api.Service{} },
    NewListFunc: func() runtime.Object { return &api.ServiceList{} },

    // ── 资源标识 ──
    DefaultQualifiedResource:  api.Resource("services"),
    SingularQualifiedResource: api.Resource("service"),

    // ── 删除行为 ──
    ReturnDeletedObject: true,

    // ── 业务规则（全部用同一个 Strategy 单例）──
    CreateStrategy:      svcreg.Strategy,
    UpdateStrategy:      svcreg.Strategy,
    DeleteStrategy:      svcreg.Strategy,
    ResetFieldsStrategy: svcreg.Strategy,
}
```

为什么 `CreateStrategy`、`UpdateStrategy`、`DeleteStrategy` 都是同一个 `svcreg.Strategy`？因为 `svcStrategy` 这个 struct 同时实现了三个接口。一个 Strategy 单例就够了。

### 第二步：创建 Status 子资源

```go
statusStore := *store                    // 复制一份
statusStore.UpdateStrategy = svcreg.StatusStrategy     // 换成 Status 专用策略
statusStore.ResetFieldsStrategy = svcreg.StatusStrategy
```

直接复制主 Store，只替换 Strategy。因为 Status 更新和主更新的区别只在"业务规则"，底层的 etcd 读写逻辑完全一样。

### 第三步：组装 REST struct 并挂载钩子

```go
genericStore := &REST{
    Store:             store,
    primaryIPFamily:   primaryIPFamily,
    secondaryIPFamily: secondaryIPFamily,
    alloc:             makeAlloc(...),  // IP + NodePort 分配器
    endpoints:         endpoints,
    pods:              pods,
    proxyTransport:    proxyTransport,
}

// 挂载钩子
store.Decorator   = genericStore.defaultOnRead   // 读取后装饰
store.AfterDelete = genericStore.afterDelete      // 删除后清理
store.BeginCreate = genericStore.beginCreate      // 创建前分配资源
store.BeginUpdate = genericStore.beginUpdate      // 更新前调整资源
```

### 四个钩子详解

**Decorator — `defaultOnRead`**

每次从 etcd 读出 Service 对象后都会调用。它做三件事：

1. 规范化 ClusterIP 和 ClusterIPs，确保它们一致
2. 为老版本对象补上新字段（IPFamilies、IPFamilyPolicy）
3. ExternalName 类型的 Service 去掉不适用的 InternalTrafficPolicy

为什么需要这个钩子？K8s 是不断演进的。一个 Service 可能在 1.20 创建（没有 IPFamilies 字段），在 1.27 被读取。Decorator 在读取时补上缺失的字段，上层代码不用处理老版本对象。这是一种**读时迁移**模式。

**BeginCreate — `beginCreate`**

创建 Service 前分配资源：

```go
func (r *REST) beginCreate(ctx, obj, options) (FinishFunc, error) {
    svc := obj.(*api.Service)

    // 1. 规范化 ClusterIP
    normalizeClusterIPs(After{svc}, Before{nil})

    // 2. 分配 ClusterIP + NodePort
    txn, err := r.alloc.allocateCreate(svc, dryrun)

    // 3. 返回事务式回调
    finish := func(ctx, success bool) {
        if success { txn.Commit() }   // 写入 etcd 成功 → 提交分配
        else       { txn.Revert() }   // 写入失败 → 回滚，释放 IP/端口
    }
    return finish, nil
}
```

为什么分配 ClusterIP 要在 `BeginCreate` 而不是 `PrepareForCreate`？因为 `PrepareForCreate` 是 Strategy 的一部分，而 Strategy 不应该依赖外部资源（如 IP 分配器）。`BeginCreate` 是 Store 层的钩子，可以访问外部资源。这是**职责分离**：Strategy 只管数据规则，Store 钩子管资源操作。

而且 `BeginCreate` 返回的 `FinishFunc` 实现了**事务语义**：如果 etcd 写入失败，IP 会被回滚。否则你会"占着 IP 但没有 Service"，造成资源泄漏。

**BeginUpdate — `beginUpdate`**

更新时最复杂的钩子，处理 Service 类型变更时的资源调整：

```go
func (r *REST) beginUpdate(ctx, obj, oldObj, options) (FinishFunc, error) {
    newSvc := obj.(*api.Service)
    oldSvc := oldObj.(*api.Service)

    // 1. 为老对象补上缺失字段
    r.defaultOnReadService(oldSvc)

    // 2. 补全用户没指定的已分配值（幂等性）
    patchAllocatedValues(After{newSvc}, Before{oldSvc})

    // 3. 分配/释放资源
    txn, err := r.alloc.allocateUpdate(After{newSvc}, Before{oldSvc}, dryrun)

    return finish, nil
}
```

`patchAllocatedValues` 保证了幂等性——当用户用 `kubectl apply` 更新 Service 时，可能没有指定 ClusterIP（因为它是自动分配的）。这个函数会把旧对象的 ClusterIP 复制到新对象，确保不丢失。

**AfterDelete — `afterDelete`**

删除 Service 后清理资源：删除关联的 Endpoints 对象，释放 ClusterIP、NodePort、HealthCheckNodePort。

为什么删除 Endpoints 要在这里做，而不是让 Endpoints Controller 自己处理？为了防止"名字复用"问题。如果 Service 被删除后立即创建同名新 Service，而旧的 Endpoints 还没被 Controller 清理，新 Service 就会错误地指向旧的后端 Pod。所以在 API 层同步删除 Endpoints 更安全。

### Strategy 中的 dropTypeDependentFields

当用户改变 Service 类型时（如 ClusterIP → ExternalName），需要清理不再适用的字段：

```go
func dropTypeDependentFields(newSvc, oldSvc *api.Service) {
    // 从需要 ClusterIP 的类型 → 不需要的类型
    if needsClusterIP(oldSvc) && !needsClusterIP(newSvc) {
        if sameClusterIPs(oldSvc, newSvc) {  // 用户没手动改
            newSvc.Spec.ClusterIP = ""
            newSvc.Spec.ClusterIPs = nil
        }
    }
    // NodePort、IPFamilies 等同理...
}
```

为什么要检查 `sameClusterIPs`（用户没手动改）才清除？只有同时满足三个条件才清除：字段不再适用、字段是自动初始化的、用户没有同时试图修改它。如果用户手动把 ClusterIP 改了（即使类型不对），应该让 Validate 去报错，而不是静默清除。这是一种**"不替用户做决定"的设计哲学**。

## 第五层：从 Store 到 HTTP 路由 — "资源注册的完整旅程"

### 全景图

```
  ▼
```

### 第一步：资源提供者组装 Storage Map

`pkg/registry/core/rest/storage_core.go` 是 core v1 组的注册入口：

```go
func (c LegacyRESTStorageProvider) NewLegacyRESTStorage(...) (LegacyRESTStorage, APIGroupInfo, error) {
    // 1. 创建 APIGroupInfo
    apiGroupInfo := genericapiserver.APIGroupInfo{
        VersionedResourcesStorageMap: map[string]map[string]rest.Storage{},
        ...
    }

    // 2. 创建各个资源的 Store
    podStorage := podstore.NewStorage(...)
    serviceRESTStorage, serviceStatusStorage, serviceProxyStorage := servicestore.NewREST(...)

    // 3. 注册到 storage map
    storage := map[string]rest.Storage{}
    storage["pods"] = podStorage.Pod
    storage["pods/status"] = podStorage.Status
    storage["pods/exec"] = podStorage.Exec
    storage["services"] = serviceRESTStorage
    storage["services/status"] = serviceStatusStorage
    storage["services/proxy"] = serviceProxyStorage

    // 4. 放入 VersionedResourcesStorageMap
    apiGroupInfo.VersionedResourcesStorageMap["v1"] = storage
}
```

子资源（`pods/status`、`pods/exec`）和主资源用同一个 map 注册，只是 key 带了 `/` 后缀。安装器会自动解析 `/` 来区分主资源和子资源。

### 第四步：registerResourceHandlers — 核心中的核心

这是整个流程最关键的一个函数（约 820 行）。

**4.1 接口断言 — "你支持哪些操作？"**

```go
creater, isCreater := storage.(rest.Creater)
lister, isLister := storage.(rest.Lister)
getter, isGetter := storage.(rest.Getter)
gracefulDeleter, isGracefulDeleter := storage.(rest.GracefulDeleter)
updater, isUpdater := storage.(rest.Updater)
patcher, isPatcher := storage.(rest.Patcher)
watcher, isWatcher := storage.(rest.Watcher)
connecter, isConnecter := storage.(rest.Connecter)
// ... 一共 13 个接口检查
```

这就是为什么第一层要拆分接口——这里用 Go 的类型断言来探测 storage 实现了哪些接口，从而决定注册哪些路由。

**4.2 构建 Action 列表**

```go
// 集合操作
actions = appendIf(actions, action{"LIST",              resourcePath, ...}, isLister)
actions = appendIf(actions, action{"POST",              resourcePath, ...}, isCreater)
actions = appendIf(actions, action{"DELETECOLLECTION",  resourcePath, ...}, isCollectionDeleter)

// 单对象操作
actions = appendIf(actions, action{"GET",    itemPath, ...}, isGetter)
actions = appendIf(actions, action{"PUT",    itemPath, ...}, isUpdater)
actions = appendIf(actions, action{"PATCH",  itemPath, ...}, isPatcher)
actions = appendIf(actions, action{"DELETE", itemPath, ...}, isGracefulDeleter)
actions = appendIf(actions, action{"WATCH",  "watch/" + itemPath, ...}, isWatcher)
```

`appendIf` 是一个巧妙的辅助函数：

```go
func appendIf(actions []action, a action, shouldAppend bool) []action {
    if shouldAppend {
        return append(actions, a)
    }
    return actions
}
```

为什么用 action 结构体而不是直接注册路由？因为 action 列表是一个**中间表示**。先构建 IR，再统一转换为路由，让代码更清晰，也方便后续添加 OpenAPI 元数据、metrics 标注等。

**4.3 Action → HTTP Route**

```go
for _, action := range actions {
    switch action.Verb {
    case "GET":
        handler := restfulGetResource(getter, reqScope)
        handler = metrics.InstrumentRouteFunc(...)
        route := ws.GET(action.Path).To(handler)...
    case "LIST":
        handler := restfulListResource(lister, watcher, reqScope, false, ...)
        route := ws.GET(action.Path).To(handler)...
    case "POST":
        handler := restfulCreateResource(creater, reqScope, admit)
        route := ws.POST(action.Path).To(handler)...
    case "PUT":
        handler := restfulUpdateResource(updater, reqScope, admit)
        route := ws.PUT(action.Path).To(handler)...
    case "PATCH":
        handler := restfulPatchResource(patcher, reqScope, admit, supportedTypes)
        route := ws.PATCH(action.Path).To(handler)...
    case "DELETE":
        handler := restfulDeleteResource(gracefulDeleter, ...)
        route := ws.DELETE(action.Path).To(handler)...
    case "WATCH":
        handler := restfulListResource(lister, watcher, reqScope, true, ...)  // forceWatch=true
        route := ws.GET("watch/" + action.Path).To(handler)...
    }
    ws.Route(route)
}
```

WATCH 路由和 LIST 路由用的是同一个 handler（`restfulListResource`），区别只是 `forceWatch` 参数。当请求带 `?watch=true` 时，LIST handler 自动切换为 watch 模式。

### Service 的完整路由注册

```
集群级别（跨命名空间）：
  GET    /api/v1/services                                    → LIST

命名空间级别：
  GET    /api/v1/namespaces/{ns}/services                    → LIST
  POST   /api/v1/namespaces/{ns}/services                    → CREATE
  DELETE /api/v1/namespaces/{ns}/services                    → DELETECOLLECTION
  GET    /api/v1/namespaces/{ns}/services/{name}             → GET
  PUT    /api/v1/namespaces/{ns}/services/{name}             → UPDATE
  PATCH  /api/v1/namespaces/{ns}/services/{name}             → PATCH
  DELETE /api/v1/namespaces/{ns}/services/{name}             → DELETE
  GET    /api/v1/watch/namespaces/{ns}/services/{name}       → WATCH

子资源：
  PUT    /api/v1/namespaces/{ns}/services/{name}/status      → UPDATE STATUS
  GET    /api/v1/namespaces/{ns}/services/{name}/proxy       → PROXY
```

