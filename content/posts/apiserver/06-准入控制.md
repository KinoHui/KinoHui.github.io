## 准入控制在请求链路中的位置

Admission 不在 filter chain（HTTP 中间件）中，而是在 filter chain 之后的 endpoint handler 内部被调用。

```
HTTP 请求进入
  │
  │  ┌─── Filter Chain（HTTP 中间件）───┐
  │  │  WithAuthentication → WithAuthorization → ... → Director  │
  │  └────────────────────────────────────────────────────────────┘
  │
  │  apiHandler = Director → GoRestfulContainer → 路由匹配
  │
  ▼  endpoint handler（create.go / update.go / patch.go / delete.go）
     │
     ├─ 解码请求 body
     ├─ 从 storage 获取对象
     ├─ admission.Admit()    ← mutating admission 在这里
     ├─ storage.Create/Update/Patch/Delete
     └─ admission.Validate() ← validating admission 在这里
```

**关键区别**：
- Filter chain：处理 HTTP 层面的通用逻辑（认证、鉴权、审计、限流、CORS），对所有请求生效
- Admission：处理业务层面的对象校验逻辑，只有在具体的 CRUD 操作 handler 内部才会被调用

## 两阶段架构

```
HTTP Request
    │
    ▼
┌─────────────────────────────────────────────┐
│  Phase 1: Mutating Admission (变更准入)      │
│  - 可以修改对象                              │
│  - 插件顺序执行（串行）                       │
│  - 包括内置插件 + MutatingWebhook            │
└─────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────┐
│  Phase 2: Validating Admission (验证准入)     │
│  - 只能拒绝，不能修改对象                     │
│  - Webhook 并行调用，内置插件串行             │
│  - 包括内置插件 + ValidatingWebhook          │
└─────────────────────────────────────────────┘
    │
    ▼
  写入 etcd
```

## 核心接口

```go
type Interface interface {
    Handles(operation Operation) bool  // 判断是否处理该操作
}

type MutationInterface interface {
    Interface
    Admit(ctx context.Context, a Attributes, o ObjectInterfaces) error  // 变更阶段，可修改对象
}

type ValidationInterface interface {
    Interface
    Validate(ctx context.Context, a Attributes, o ObjectInterfaces) error  // 验证阶段，只能拒绝
}
```

同一个插件可以同时实现两个接口（比如 LimitRanger、ResourceQuota），这样它在两个阶段都会被调用。

## 插件执行顺序

`AllOrderedPlugins` 定义了全局唯一的执行顺序：

```
...（约 30 个 in-tree mutating 插件）...
MutatingAdmissionWebhook      ← 倒数第 4
ValidatingAdmissionPolicy     ← 倒数第 3
ValidatingAdmissionWebhook    ← 倒数第 2
ResourceQuota                 ← 倒数第 1（需要看到最终对象状态）
AlwaysDeny
```

**为什么顺序很重要**：
- Mutating webhook 放最后，确保所有内置插件的变更已完成，webhook 看到的是"几乎最终"的对象
- ResourceQuota 放验证 webhook 之后，确保只有通过所有验证的对象才计入配额

## Chain 构建（初始化时）

`NewFromPlugins` 的处理流程：

1. 按 `AllOrderedPlugins` 顺序实例化每个插件
2. 收集到 `[]Interface` 切片
3. 包装为 `chainAdmissionHandler`（顺序执行）
4. 包装为 `reinvoker`（支持重调）
5. 包装为 `WithStepMetrics`（记录延迟指标）

最终的包装层次（从外到内）：

```
WithStepMetrics          ← 指标收集
  └─ WithAudit          ← 审计注解
      └─ ManagedFields  ← managed fields 校验
          └─ reinvoker  ← 重调控制
              └─ chainAdmissionHandler ← 顺序遍历所有插件
                  ├─ 插件1（NamespaceLifecycle）
                  ├─ 插件2（LimitRanger）
                  ├─ ...
                  ├─ MutatingAdmissionWebhook
                  ├─ ValidatingAdmissionPolicy
                  ├─ ValidatingAdmissionWebhook
                  └─ ResourceQuota
```

## chainAdmissionHandler 的执行逻辑

```go
func (admissionHandler chainAdmissionHandler) Admit(ctx, a, o) error {
    for _, handler := range admissionHandler {
        if !handler.Handles(a.GetOperation()) {
            continue                    // 不处理该操作，跳过
        }
        if mutator, ok := handler.(MutationInterface); ok {
            err := mutator.Admit(ctx, a, o)
            if err != nil {
                return err              // 遇到第一个错误就返回（fail fast）
            }
        }
    }
    return nil
}
```

## 准入链是全局共享的

不是每个版本一条链，而是整个 server 共享一条链：

```
s.admissionControl  (server 启动时构建的一条全局链)
    │
    ├──→ APIGroupVersion{group: "apps",  version: "v1"}  .Admit = s.admissionControl
    ├──→ APIGroupVersion{group: "",      version: "v1"}  .Admit = s.admissionControl
    ├──→ APIGroupVersion{group: "batch", version: "v1"}  .Admit = s.admissionControl
    └──→ ...所有 APIGroupVersion 共享同一个实例
```

## Create 请求中的完整准入流程

```
HTTP POST /api/v1/namespaces/default/pods
    │
    ▼
createHandler()
    │
    ├── 1. 解析 namespace/name
    ├── 2. 反序列化 body → obj
    ├── 3. 构建 admissionAttributes
    │
    ▼
finisher.FinishRequest()
    │
    ▼
┌─────────────────────────────────────────────────────────┐
│  Phase 1: Mutating Admit                                │
│  mutatingAdmission.Admit(ctx, attributes, scope)         │
│  → 执行 reinvoker.Admit()                               │
│    → chainAdmissionHandler.Admit() 串行遍历所有插件       │
│    → 每个插件可修改 obj                                   │
└─────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────┐
│  Phase 2: Validating (通过 requestFunc 传入存储层)        │
│  requestFunc = func() {                                  │
│      r.Create(ctx, name, obj,                            │
│          rest.AdmissionToValidateObjectFunc(admit, ...),  │
│          options)                                         │
│  }                                                       │
│  存储层在真正写 etcd 之前调用这个 ValidateObjectFunc       │
└─────────────────────────────────────────────────────────┘
    │
    ▼
  写入 etcd，返回 201 Created
```

**Validate 进入存储层的方式**（闭包传递）：

为什么用闭包？因为 Validate 需要在 Mutating 之后、etcd 写入之前执行。通过把 Validate 包装成闭包传入存储层，存储层可以在自己内部的合适时机调用它。

---

## 各操作的准入调用点对比

| 操作 | Mutating 调用点 | Validating 调用点 |
|------|----------------|------------------|
| Create | finisher 闭包内，r.Create() 之前 | r.Create() 内部，写 etcd 之前 |
| Update | finisher 闭包内，r.Update() 之前 | r.Update() 内部，写 etcd 之前 |
| Patch | applyPatch() 内，patch 计算之后 | r.Update() 内部 |
| Delete | 无 Mutating（Delete 通常不改对象） | r.Delete() 内部，删 etcd 之前 |

---

## 一句话总结

整个准入链从注册、构建到执行的完整流程。核心设计思想：mutating chain 支持最多两轮执行（第二轮按需触发），validating chain 只执行一轮。
