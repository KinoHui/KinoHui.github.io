## 从抽象到具体

前面看了注册链路的框架，现在从"具体的实现"往上看。先看一个最简单的 Storage 实现，理解它怎么和框架对接。

---

## genericregistry.Store — 通用 CRUD 引擎

`genericregistry.Store` 是 `rest.Storage` 接口的通用实现，几乎所有 K8s 资源都复用它：

```
genericregistry.Store 实现了：
  ├── rest.Storage          (New, Destroy)
  ├── rest.Lister           (List, NewList)
  ├── rest.Getter           (Get)
  ├── rest.Creater          (Create)
  ├── rest.Updater          (Update)
  ├── rest.GracefulDeleter  (Delete)
  ├── rest.CollectionDeleter(DeleteCollection)
  ├── rest.Watcher          (Watch)
  ├── rest.Scoper           (NamespaceScoped)
  └── ... 全部接口
```

---

## Object 接口 — 所有 API 对象的根接口

```go
type Object interface {
    GetObjectKind() schema.ObjectKind  // 获取对象的 GVK 信息
    DeepCopyObject() Object            // 深拷贝对象
}
```

Pod、Service、Deployment、ConfigMap... 全部都必须实现这个接口，它是 K8s 序列化、深拷贝、识别资源类型的基础。

- `GetObjectKind()` = 身份证 — 一看就知道你是 Deployment、Pod 还是 Service
- `DeepCopyObject()` = 复印机 — 需要修改时，先复印一份，不影响原件

---

## Storage 接口重构计划

Kubernetes 计划把存储操作拆成两大组：

**集合操作接口**（操作一类资源整体）：

| 方法 | 旧名字 | 新名字 | 作用 |
|------|--------|--------|------|
| GET | Lister | CollectionGetter | 获取资源列表 |
| WATCH | Watcher | CollectionWatcher | 监听资源列表变化 |
| CREATE | Creater | CollectionCreater | 在集合里创建一个资源 |
| DELETE | 无 | CollectionDeleter | 批量删除 |

**单个资源接口**（操作某个具体命名的资源）：

| 方法 | 旧名字 | 新名字 | 作用 |
|------|--------|--------|------|
| GET | Getter | NamedGetter | 获取单个命名资源 |
| DELETE | Deleter | NamedDeleter | 删除指定名称的资源 |
| UPDATE | Update | NamedUpdater | 更新指定名称的资源 |

## LimitRange — 一个最简单的 Storage 实现

`pkg/registry/core/limitrange/` 目录总共就 3 个文件：
- `storage/storage.go` — 实现 rest.Storage 接口
- `strategy.go` — 定义 CRUD 策略（校验、默认值）
- `doc.go`

### storage/storage.go — 把一切组装起来

```go
type REST struct {
    *genericregistry.Store   // 嵌入通用 Store，继承所有 CRUD 能力
}
```

核心就一个 `NewREST` 函数，配置一个 `genericregistry.Store`：

```go
store := &genericregistry.Store{
    NewFunc:     func() runtime.Object { return &api.LimitRange{} },       // 空对象
    NewListFunc: func() runtime.Object { return &api.LimitRangeList{} },   // 空列表对象
    DefaultQualifiedResource:  api.Resource("limitranges"),   // 资源名
    SingularQualifiedResource: api.Resource("limitrange"),    // 单数名
    CreateStrategy: limitrange.Strategy,   // ← 创建时的校验逻辑
    UpdateStrategy: limitrange.Strategy,   // ← 更新时的校验逻辑
    DeleteStrategy: limitrange.Strategy,   // ← 删除时的校验逻辑
}
```

LimitRange 注册出来的路由是完整的：

```
GET    /api/v1/namespaces/{ns}/limitranges            ← List
POST   /api/v1/namespaces/{ns}/limitranges            ← Create
GET    /api/v1/namespaces/{ns}/limitranges/{name}     ← Get
PUT    /api/v1/namespaces/{ns}/limitranges/{name}     ← Update
PATCH  /api/v1/namespaces/{ns}/limitranges/{name}     ← Patch
DELETE /api/v1/namespaces/{ns}/limitranges/{name}     ← Delete
DELETE /api/v1/namespaces/{ns}/limitranges            ← DeleteCollection
GET    /api/v1/namespaces/{ns}/limitranges?watch=true ← Watch
```

### strategy.go — 定义业务规则

```go
type limitrangeStrategy struct {
    runtime.ObjectTyper     // 能判断对象的 GVK
    names.NameGenerator     // 能生成名字
}
var Strategy = limitrangeStrategy{legacyscheme.Scheme, names.SimpleNameGenerator}
```

关键方法：

| 方法 | 作用 | LimitRange 的实现 |
|------|------|-------------------|
| `NamespaceScoped()` | 是否是命名空间级资源 | `return true` |
| `PrepareForCreate()` | 创建前的数据准备 | 如果没名字就自动生成 UUID |
| `PrepareForUpdate()` | 更新前的数据准备 | 空（不需要特殊处理） |
| `Validate()` | 创建时的校验 | 调用 `validation.ValidateLimitRange` |
| `ValidateUpdate()` | 更新时的校验 | 同上 |
| `AllowCreateOnUpdate()` | PUT 时对象不存在能否创建 | `return true` |
| `AllowUnconditionalUpdate()` | 能否不带 resourceVersion 更新 | `return true` |

### 两个文件的关系

```
strategy.go                          storage/storage.go
───────────                          ──────────────────
定义"规则"                           定义"存储"

limitrangeStrategy {                 REST {
  NamespaceScoped → true               *genericregistry.Store ← 实现了所有 CRUD
  PrepareForCreate → 自动生成名          │
  Validate → 校验规则                    │  配置 Store 时引用 Strategy：
}                                       │    CreateStrategy = Strategy
                                        │    UpdateStrategy = Strategy
                                        │    DeleteStrategy = Strategy
                                      }

   你负责"怎么校验"                    我负责"怎么存取"
```

**设计模式**：`genericregistry.Store` 处理所有资源的通用逻辑（序列化、etcd 读写、watch、分页），Strategy 只关心这个资源独有的业务规则。Kubernetes 里几乎每个资源都是这个模式。

## 为什么需要深拷贝？

Kubernetes 的设计规则：**所有缓存里的对象、etcd 里读出来的对象，一律只读！不能改！**

因为同一个对象会被很多地方同时使用：控制器、调度器、客户端请求、缓存、监听。它们共用同一个内存对象。直接改原对象 = 所有人同时看到被你改掉的数据。

正确的修改流程：
1. 从缓存拷贝一份出来（深拷贝）
2. 修改这份拷贝
3. 把拷贝提交保存到 etcd
4. 系统用新副本替换旧的

**原对象只读，修改必拷贝；改完再提交，安全不会乱。**
