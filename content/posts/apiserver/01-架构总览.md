## 整体架构：委托链模式

整个 API Server 通过委托模式连接三个子 server 以及一个 404 handler，具体顺序：

```
aggregate -> apiserver -> extension -> 404
```

每个 server 在初始化 `New` 时都会初始化一个 `GenericAPIServer`，包含两种路由：
- **go-restful 路由**：K8s 主要资源路由，如 `/api/v1/`
- **PathRecorderMux 路由**：自定义路由，如 `/healthz`

初始化后调用 `installAPI(s, c.Config)` 注册基础路由：
- `/`、`/debug/*`、`/metrics`、`/version`、`/apis`
- `/logs`、`/openapi`、`/api/*`、`/apis/*`

## 三个关键组件

### serviceResolver — 内部 DNS

**作用**：把 Service 名字解析成真实的集群内部地址/IP。

APIServer 经常需要访问 `kubernetes.default.svc`、`metrics-server.kube-system.svc` 等内部服务，但这些是服务名而非真实地址。`serviceResolver` 负责完成这个转换。

它被传递给：
- API 扩展服务器（CRD）— 因为需要访问集群内部服务
- 聚合层（Aggregator）— 因为需要调用其他 APIService

### pluginInitializer — 依赖注入器

**作用**：给各种插件注入依赖（授权、配置、客户端等）。

K8s APIServer 的扩展功能都是插件（认证插件、授权插件、准入控制插件、云厂商插件等），它们需要：
- kube-apiserver 的配置
- 内部客户端
- 共享 informer
- 认证信息
- serviceResolver

`pluginInitializer` 把这些核心能力统一打包，自动注入给所有插件。

### NewAPIServerHandler — 统一请求入口

定义在 `vendor/k8s.io/apiserver/pkg/server/handler.go`，创建 `APIServerHandler` 结构体：

```
请求 → FullHandlerChain（拦截链）
        ↓
      Director（总指挥）
      ↙          ↘
go-restful      nonGoRestful
(API接口)        (健康检查等)
      ↓              ↓
  处理请求        处理请求
      ↘              ↙
       都不行 → 404 委托处理器
```

关键字段：
- `FullHandlerChain`：完整过滤器链，绑在 HTTPS 端口上
- `GoRestfulContainer`：go-restful 的路由容器，处理 `/apis/...`
- `NonGoRestfulMux`：处理 `/healthz`、`/metrics` 等非 REST 路由
- `Director`：决策者，判断请求走 GoRestful 还是 NonGoRestful

核心赋值：
```go
FullHandlerChain: handlerChainBuilder(director)
```
把 director（总指挥）套上一层完整的请求拦截链（审计、认证、鉴权、限流、Panic 恢复、请求日志等）。

---

## Director 的路由决策

定义在 `vendor/k8s.io/apiserver/pkg/server/handler.go`，Director 根据 URL 路径决定走哪条路由：

- `/apis` 或 `/apis/` 精准匹配 → go-restful 路由
- `/api/xxxx` 或 `/api/xxxx/` 前缀匹配 → go-restful 路由
- `/healthz`、`/metrics` 等 → nonGoRestful 路由

## 启动三阶段

GenericAPIServer 结构体保存了所有关键运行时状态：`admissionControl`、`Authorizer`、`AuditBackend`、请求跟踪的 WaitGroup，以及处理程序架构的引用。

### 阶段一：装配

构建 GenericAPIServer，插入所有核心组件。

### 阶段二：PrepareRun()

服务器还没有开始接受请求，做"开门前的准备工作"：
- 注册 `/healthz`、`/readyz`、`/livez` 健康检查端点
- 注册 PostStart Hooks（启动后执行的回调，如启动 informer、同步缓存）
- 注册 `/openapi/v2`、`/openapi/v3` 端点

### 阶段三：Run(ctx)

启动安全的 HTTPS 服务器，开始监听端口并接受连接。`ctx` 用于优雅关闭 — 当上下文取消时，服务器等待已有请求处理完再退出。

**一句话总结**：先攒零件（装配）→ 再做准备（PrepareRun）→ 最后开跑（Run）。经典的"构建 → 初始化 → 启动"三段式。
