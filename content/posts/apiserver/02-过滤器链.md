## 核心思路：洋葱模型

过滤器链本质是嵌套的 `http.Handler`，每个过滤器把下一层 handler 包裹起来，形成洋葱结构。请求从外层进去，响应从内层出来。

```
请求进入
    │
    ▼
┌─────────────────────────────────┐
│  WithPanicRecovery              │  ← 最外层，兜底
│  ┌───────────────────────────┐  │
│  │  WithRetryAfter           │  │  ← 关闭期间返回 503
│  │  ┌─────────────────────┐  │  │
│  │  │  WithHTTPLogging    │  │  │  ← 记录请求日志
│  │  │  ┌───────────────┐  │  │  │
│  │  │  │  WithAudit    │  │  │  │  ← 审计日志
│  │  │  │  ┌─────────┐  │  │  │  │
│  │  │  │  │ WithAuth│  │  │  │  │  ← 认证 + 授权
│  │  │  │  │  ...    │  │  │  │  │
│  │  │  │  │ Director│  │  │  │  │  ← 最内层，真正分发
│  │  │  │  └─────────┘  │  │  │  │
│  │  │  └───────────────┘  │  │  │
│  │  └─────────────────────┘  │  │
│  └───────────────────────────┘  │
└─────────────────────────────────┘
```

## 装配入口

入口在 `config.go`：

```go
handlerChainBuilder := func(handler http.Handler) http.Handler {
    return c.BuildHandlerChainFunc(handler, c.Config)
}
```

`BuildHandlerChainFunc` 是一个可替换的函数指针（Kubernetes 留的定制口子），默认实现是 `DefaultBuildHandlerChain`。它接收最内层的 Director handler，然后一层层往外包。

**关键认知**：代码是由内向外包裹的，所以代码写在前面的 = 请求处理时靠后执行。读这个函数要从下往上读，才是请求真正经历的顺序。

## 完整调用链（按请求实际经历的顺序，从外到内）

```
请求
 │
 ▼
[1]  WithAuditInit              ← 初始化审计 context（最外层）
[2]  WithPanicRecovery          ← recover panic，防止进程崩溃
[3]  WithMuxAndDiscoveryComplete← 等待路由注册完成才放行
[4]  WithRequestReceivedTimestamp← 打上请求接收时间戳
[5]  WithRequestInfo            ← 解析请求路径，填充 RequestInfo（verb/resource/namespace 等）
[6]  WithRoutine (可选)         ← 在独立 goroutine 里执行后续 handler，减少栈内存占用
[7]  WithLatencyTrackers        ← 初始化各阶段延迟追踪
[8]  WithHTTPLogging            ← 记录 HTTP 请求/响应日志
[9]  WithRetryAfter (可选)      ← 服务关闭期间返回 503 + Retry-After
[10] WithHSTS                   ← 写入 Strict-Transport-Security 响应头
[11] WithCacheControl           ← 写入 Cache-Control: no-cache 响应头
[12] WithProbabilisticGoaway (可选) ← HTTP/2 随机发送 GOAWAY，强制客户端重连（负载均衡）
[13] WithWatchTerminationDuringShutdown (可选) ← 关闭期间优雅终止 watch 请求
[14] WithWaitGroup              ← 非长运行请求注册进 WaitGroup，优雅关闭用
[15] WithRequestDeadline        ← 根据请求类型设置 context deadline
[16] WithTimeoutForNonLongRunningRequests ← 非长运行请求开 goroutine + 超时计时器
[17] WithWarningRecorder        ← 注入 warning header 记录器（线程安全）
[18] WithCORS                   ← 处理跨域请求头
[19] WithAuthentication         ← 【认证】识别请求者身份，失败返回 401
[20] WithTracing                ← 认证后启动 trace span（认证用户可影响采样策略）
[21] WithAudit                  ← 写审计日志（认证后才有用户信息）
[22] WithImpersonation /
     WithConstrainedImpersonation ← 处理 impersonate 头，允许用户扮演其他用户
[23] WithPriorityAndFairness /
     WithMaxInFlightLimit        ← 【流量控制】APF 或最大并发数限制
[24] WithAuthorization          ← 【授权】判断用户是否有权限，失败返回 403
[25] apiHandler (Director)      ← 真正的业务分发（最内层）
```

## 每个过滤器的通用模式

以认证过滤器为例：

```go
func WithAuthentication(handler http.Handler, auth authenticator.Request, ...) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, req *http.Request) {
        resp, ok, err := auth.AuthenticateRequest(req)
        if err != nil || !ok {
            http.Error(w, "Unauthorized", 401)
            return  // ← 短路，不调用 handler.ServeHTTP
        }
        // 把用户信息存入 context，传递给下一层
        req = req.WithContext(genericapirequest.WithUser(req.Context(), resp.User))
        handler.ServeHTTP(w, req)  // ← 通过，继续往里走
    })
}
```

**关键模式**：
- **失败** → 直接写响应，不调用 `handler.ServeHTTP`，链路在此短路
- **成功** → 把结果存入 context，调用 `handler.ServeHTTP` 继续传递

这就是为什么内层 handler 可以通过 `ctx` 拿到认证用户、RequestInfo 等信息 — 都是外层过滤器塞进去的。

## 值得注意的细节

### 1. 延迟追踪：TrackStarted / TrackCompleted 成对出现

```go
handler = filterlatency.TrackCompleted(handler)
handler = genericapifilters.WithAuthorization(handler, ...)
handler = filterlatency.TrackStarted(handler, c.TracerProvider, "authorization")
```

每个关键过滤器前后都夹着延迟追踪。`TrackStarted` 记录进入时间，`TrackCompleted` 记录退出时间，两者配合算出每层耗时，暴露到 metrics。

**配对关系**（以认证过滤器为例）：

```
TrackStarted("authentication")   ← 开始计时，写入 context
        └── WithAuthentication   ← 被计时的目标 filter
                └── TrackCompleted ← 结束计时，读 context 算差值上报 metrics
```

`TrackCompleted` 的时间戳打在"进入时"而非"退出时" — 它记录的是"上一个 filter 刚刚执行完毕、即将交棒给我的那一刻"。两者通过 context 握手，中间夹的那个 filter 就是被测量的对象。

### 2. 认证失败走独立的 failedHandler（分叉机制）

```go
failedHandler := genericapifilters.Unauthorized(c.Serializer)
failedHandler = genericapifilters.WithFailedAuthenticationAudit(failedHandler, c.AuditBackend, ...)
handler = genericapifilters.WithAuthentication(handler, ..., failedHandler, ...)
```

`WithAuthentication` 同时持有两个 handler 引用，根据认证结果决定调用哪一个：

```
                    WithAuthentication
                           │
               auth.AuthenticateRequest(req)
                           │
              ┌────────────┴────────────┐
           失败(err/!ok)              成功
              │                        │
     failed.ServeHTTP(w, req)    handler.ServeHTTP(w, req)
              │                        │
   ┌──────────┴──────────┐      后续过滤器链继续...
   WithFailedAuthAudit   │      (Impersonation → APF →
   (写审计日志)           │       Authorization → Director)
              │
         Unauthorized
         (返回 401)
```

### 3. APF 和 MaxInFlight 二选一

```go
if c.FlowControl != nil {
    handler = genericfilters.WithPriorityAndFairness(...)  // 现代模式
} else {
    handler = genericfilters.WithMaxInFlightLimit(...)      // 简单模式
}
```

`FlowControl != nil` 时启用 Priority-and-Fairness（APF），这是 1.20+ 的默认行为，支持按优先级和公平性分配请求配额。否则退化为简单的并发数上限控制。

### 4. WithRoutine 的栈内存优化

```go
if c.FeatureGate.Enabled(genericfeatures.APIServingWithRoutine) {
    handler = routine.WithRoutine(handler, c.LongRunningFunc)
}
```

这个 filter 把后续 handler 放到新 goroutine 里执行，当前 goroutine 只做服务，目的是减少调用栈深度带来的内存占用。必须在 `WithPanicRecovery` 之内，否则新 goroutine 的 panic 无法被捕获。

### 5. WithTracing 在认证之后

```go
// WithTracing comes after authentication so we can allow authenticated
// clients to influence sampling.
```

认证用户可以通过请求头影响 trace 的采样策略，所以 tracing 必须在认证之后初始化。

## 响应从内层出来时的后处理

`handler.ServeHTTP(w, req)` 执行完，意味着所有写响应的逻辑都已经完成（数据已经写入 `w`），但此时响应还在内核缓冲区里，还没有真正发到客户端。

`w` 是 `http.ResponseWriter` 接口，filter 链里所有人共享同一个 `w`，内层 handler 写响应时直接操作的就是这个对象。所以响应的写入是穿透整个链的，不存在"响应从内层一层层往外冒"这种过程。

**有后处理能力的 filter**（在 `handler.ServeHTTP` 之后还有代码，或用了 defer）：

| Filter | 后处理内容 |
|--------|-----------|
| TrackCompleted | `defer handler.ServeHTTP`，自己的逻辑反而在前面先跑 |
| WithPanicRecovery | `defer` 捕获 panic，响应写完后兜底 |
| WithHTTPLogging | 内部用 ResponseWriter 包装器，响应结束后记录状态码、耗时 |
| WithWaitGroup | `defer wg.Done()`，响应结束后释放 WaitGroup |
| TrackStarted | `defer span.End()`，响应结束后关闭 trace span |

**没有后处理的 filter**（`handler.ServeHTTP` 是最后一行）：

| Filter | 说明 |
|--------|------|
| WithAuthentication | 调完 `handler.ServeHTTP` 直接结束 |
| WithAuthorization | 同上 |
| WithCORS | 同上 |
| WithRequestInfo | 同上 |

---

## 一句话总结

过滤器链 = 责任链模式（Chain of Responsibility）+ context 传参。每层只干一件事，干完要么短路返回错误，要么把结果塞进 context 然后调用下一层。最内层的 Director 才是真正处理业务逻辑的地方。
