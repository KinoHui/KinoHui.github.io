## 3. namespace 与 Pod 共享模型

判断两个进程是否共享某类 namespace：

```bash
sudo readlink /proc/<pid>/ns/net
sudo readlink /proc/<pid>/ns/pid
sudo readlink /proc/<pid>/ns/mnt
```

规则：

```text
inode ID 相同，说明共享对应 namespace。
```

默认 Pod：

```text
network namespace：共享
PID namespace：不共享
mount namespace：不共享
cgroup namespace：通常不共享
```

开启：

```yaml
shareProcessNamespace: true
```

后：

```text
同 Pod 内容器共享 PID namespace。
仍然不共享 mount namespace。
```

pause/sandbox 容器作用：

```text
持有 Pod sandbox，尤其是 Pod network namespace。
CNI 为 Pod network namespace 配置 IP、路由、veth。
业务容器加入 pause/sandbox 的 network namespace。
```

所以：

```text
同 Pod 内容器共享 Pod IP、localhost、端口空间。
sidecar 可以通过 127.0.0.1 访问 web 容器。
```

排障命令：

```bash
sudo nsenter -t <pid> -n ip addr
sudo nsenter -t <pid> -n ip route
sudo nsenter -t <pid> -n ss -lntp
```

意义：

```text
使用宿主机上的 ip/ss 工具，进入目标进程的 network namespace。
即使容器镜像里没有 ip/ss，也能查看 Pod netns 网络视图。
```

## 4. cgroup v2

核心对应关系：

```text
CPU limit
  -> cpu.max

CPU request
  -> cpu.weight

CPU throttling
  -> cpu.stat: nr_throttled / throttled_usec

Memory limit
  -> memory.max

Memory 当前用量
  -> memory.current

Memory OOM
  -> memory.events: oom / oom_kill

进程/线程数量限制
  -> pids.max

当前进程/线程数量
  -> pids.current
```

`cpu.max` 示例：

```text
50000 100000
```

含义：

```text
每 100000 微秒周期里最多运行 50000 微秒。
等于 0.5 core，也就是 500m。
```

`cpu.weight`：

```text
和 CPU request 相关。
不是硬限制。
在 CPU 竞争时影响分配权重。
```

`memory.max`：

```text
memory.max = max
  -> 没有内存硬限制。

memory.max = 134217728
  -> memory limit 是 128Mi。
```

`memory.current`：

```text
cgroup 视角的当前内存使用量。
不是单个进程的 VmRSS。
```

`memory.events`：

```text
oom：
当前 cgroup 内发生 OOM 的次数。

oom_kill：
当前 cgroup 内因为 OOM 触发 kill 的次数。
```

注意：

```text
oom_kill 不是整机 OOM kill 次数。
整机 OOM 要结合 dmesg / journalctl -k 判断。
```

资源性质：

```text
CPU 是可压缩资源，超限通常 throttling，表现为变慢。
Memory 是不可压缩资源，超限通常 OOM kill，进程死亡。
```

## 5. rootfs 与 overlayfs

容器没有独立内核：

```text
容器共享宿主机内核。
容器里的 / 来自运行时准备的 rootfs。
```

overlayfs 模型：

```text
lowerdir：
只读镜像层。

upperdir：
容器可写层。

workdir：
overlayfs 工作目录。

merged/rootfs：
lowerdir + upperdir 合并后，容器看到的文件系统视图。
```

三种行为：

```text
新建文件：
直接写入 upperdir。

修改 lowerdir 已有文件：
发生 copy-up，复制到 upperdir 后修改 upperdir 副本。

删除 lowerdir 已有文件：
不能删除只读 lowerdir，于是在 upperdir 创建 whiteout 标记。
merged 视图中该文件被隐藏。
```

whiteout 实验：

```text
lowerdir 中存在 /etc/nsswitch.conf。
容器里 rm /etc/nsswitch.conf。
upperdir 中出现：
c--------- root root 0,0 nsswitch.conf
```

含义：

```text
c 0,0 是 overlayfs whiteout 标记。
说明 lowerdir 中的 nsswitch.conf 在 merged 视图中被隐藏。
```

containerd snapshotter 角色：

```text
管理镜像只读层和容器可写层。
准备 overlayfs snapshot。
组织 lowerdir / upperdir / workdir。
供 runc 作为容器 rootfs 使用。
```

## 6. containerd / shim / runc / OCI

链路：

```text
kubelet
  -> CRI
  -> containerd
  -> containerd-shim-runc-v2
  -> runc
  -> Linux kernel
  -> 容器进程
```

分工：

```text
containerd：
高级容器运行时。
管理镜像、snapshot、metadata、CRI plugin、OCI bundle、runtime 调用。

containerd-shim-runc-v2：
长期托管容器进程。
让容器和 containerd 主进程解耦。
负责 stdio、退出状态、task 状态汇报等。

runc：
低级 OCI runtime。
读取 config.json。
创建 namespace、cgroup、mount。
启动容器进程。
创建完成后通常退出。
```

OCI runtime bundle：

```text
config.json
rootfs/
```

`config.json` 描述：

```text
process.args
root.path
mounts
linux.namespaces
linux.resources
capabilities
seccomp
rlimits
user
cwd
env
```

namespace 配置：

```text
带 path：
加入指定路径对应的已有 namespace。

不带 path：
由 runtime 创建新的 namespace。
```

业务容器常见：

```text
pid / mount / cgroup：
不带 path，新建。

network / ipc / uts：
带 path，加入 pause/sandbox 对应 namespace。
```

`root.path = "rootfs"`：

```text
指向 bundle 内的容器 rootfs。
这个 rootfs 通常由 containerd snapshotter 基于 overlayfs snapshot 准备。
```

## 7. kubelet 创建 Pod 调用链

高层链路：

```text
PodSpec
  -> scheduler bind 到 Node
  -> kubelet watch 到 Pod
  -> podWorkers 处理 Pod
  -> kubelet SyncPod
  -> kubeGenericRuntimeManager.SyncPod
  -> RunPodSandbox
  -> PullImage
  -> CreateContainer
  -> StartContainer
  -> containerd / shim / runc
  -> Linux 进程
```

Event 对应：

```text
Scheduled：
scheduler 把 Pod 绑定到 Node。

Pulling：
kubelet 调用 CRI ImageService.PullImage。

Pulled：
镜像拉取完成。

Created：
CRI CreateContainer 成功。
containerd 准备 metadata、snapshot、rootfs、OCI bundle。

Started：
CRI StartContainer 成功。
runc 创建 namespace/cgroup/mount 并启动进程。
shim 托管容器进程。
```

RunPodSandbox：

```text
创建 Pod sandbox / pause。
runtime 与 CNI 协作创建和配置 Pod network namespace。
CNI 配置 Pod IP、路由、veth。
pause 持有 Pod network namespace。
```

CreateContainer：

```text
准备容器 metadata。
准备 snapshot/rootfs。
准备 OCI bundle/config.json。
但容器进程不一定已经运行。
```

StartContainer：

```text
启动容器进程。
runc 按 config.json 创建 namespace/cgroup/mount。
containerd-shim 托管容器进程。
```

StartContainer 成功后，节点上能看到：

```text
业务容器进程 PID
containerd-shim 进程
/proc/<pid>/ns/*
/proc/<pid>/cgroup
/sys/fs/cgroup/...
/run/containerd/.../<cid>/config.json
/run/containerd/.../<cid>/rootfs
/var/lib/containerd/.../overlayfs/snapshots/...
```

## 9. 第 24 题标准表达

题目：

```text
请完整描述一次 Pod 从 kubelet SyncPod 到 Linux 进程运行的大致链路。
```

建议答案：

```text
kubelet watch 到被调度到本节点的 Pod。
podWorkers 接收该 Pod 的更新，并串行同步这个 Pod。
kubelet SyncPod 准备 volume、secret、configmap 等。
然后调用 runtime manager 的 SyncPod。
runtime manager 判断 Pod sandbox 是否存在，是否需要重建。
如果需要，调用 CRI RunPodSandbox。
containerd 创建 Pod sandbox / pause，CNI 配置 Pod network namespace、Pod IP、路由、veth。
pause 持有 Pod network namespace。
然后 kubelet 调用 PullImage 拉取业务容器镜像。
再调用 CreateContainer，containerd 准备 metadata、snapshot、rootfs、OCI bundle/config.json。
随后调用 StartContainer，containerd-shim 启动 task，runc 读取 config.json。
runc 创建或加入指定 namespace，设置 cgroup，准备 mount/rootfs，启动容器进程。
runc 退出，containerd-shim 长期托管容器进程。
kubelet 通过 CRI 观察容器状态并更新 PodStatus。
最终节点上可以看到 pause 进程、业务容器进程、shim 进程、/proc namespace、cgroup、OCI bundle 和 overlayfs snapshot。
```

## 10. 查漏补缺清单

### cgroup memory

```bash
cat /sys/fs/cgroup/<path>/memory.max
cat /sys/fs/cgroup/<path>/memory.current
cat /sys/fs/cgroup/<path>/memory.events
```

记忆：

```text
memory.max 是内存硬限制。
memory.current 是 cgroup 当前内存使用量。
oom / oom_kill 是当前 cgroup 视角。
```

### CPU throttling

```bash
cat /sys/fs/cgroup/<path>/cpu.max
cat /sys/fs/cgroup/<path>/cpu.stat
```

记忆：

```text
nr_throttled
throttled_usec
```

### overlayfs 删除 lowerdir 文件

```text
删除 lowerdir 文件：
不会复制文件。
会在 upperdir 创建 whiteout。
whiteout 让 merged 视图隐藏 lowerdir 文件。
```

### StartContainer 后 Linux 证据

```bash
pstree -aps <pid>
readlink /proc/<pid>/ns/net
cat /proc/<pid>/cgroup
ls /run/containerd/io.containerd.runtime.v2.task/k8s.io/<cid>
cat /run/containerd/io.containerd.runtime.v2.task/k8s.io/<cid>/config.json
```
